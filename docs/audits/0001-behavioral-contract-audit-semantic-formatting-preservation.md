# Investigation report: behavioral contract audit, semantic & formatting preservation (ch. 2, 21)

Detailed findings behind `docs/adr/0023-behavioral-contract-audit-semantic-formatting-preservation.md`. `docs/behavioral_contract.md` chapters **2** (Preservation of Source Semantics) and **21** (Formatting and Source Preservation) were checked line-by-line against every per-check autofix module: `forbid_vars.py`, `excessive_blank_lines.py`, `misplaced_comment.py`, `redundant_assignment/autofix.py` (plus `analysis.py`/`semantic.py`, which feed it), and `validate_function_name/autofix.py` — `redundant_super_init.py` has no autofix at all (`fix()` always returns `False`), so neither chapter's fixer-specific items apply to it, matching `0011`'s identical scoping note. Sequenced after `0011`'s fix-engine write/validation audit per issue #39, following on from `0011`–`0022`'s prior audits, per issue #30/#36.

Unlike the prior write-path audit (`0011`), which mostly confirmed existing behavior, this one reproduced twenty-five genuine, previously-untested defects (plus one further performance-only gap) — each demonstrated end-to-end against a real file before being fixed, not merely reasoned about in the abstract. All but two are in `forbid_vars`'s closure-following rename, added by this same audit: correctly renaming a variable across arbitrary nested scopes requires modeling substantially more of Python's actual scope/evaluation-time semantics than a first pass tends to assume, and each defect below exposed a different way the traversal's assumptions didn't match those semantics — which constructs bind a name; which parts of a binding construct's own syntax still run in a _different_ scope than the one it introduces; whether a rename's own suggested replacement name can collide with something else once closure-following actually applies it; whether the uniqueness check sees every kind of binding, and whether the shadow check itself stays within the scope it claims to; whether the reserved-name set accounts for a name only ever mentioned via `global`/`nonlocal`, and whether that check stays cheap enough to call once per violation instead of becoming quadratic; whether suggestion-assignment order reflects every scope closure-following can actually reach rather than just how the AST happens to be laid out in the source; whether "evaluated in the enclosing scope" still holds for every kind of syntax the outer/own-scope split treats that way, under every module-level compiler directive and lazily-evaluated construct that can change when, or against what, a piece of syntax is actually evaluated; whether a lazily-evaluated construct can bind names visible only to its own peers, rather than always resolving through the enclosing scope; whether the mechanism that starts a rename _within_ a given scope stays confined to the syntax that actually runs inside that scope, rather than reaching into that same scope's own outer-evaluated syntax and applying that scope's mapping to text that actually runs in whichever scope contains it; whether every construct with its own implicit type-parameter scope — not just a generic function — gets the same peer-aware treatment; whether a scope-boundary split built around only two buckets (evaluated in the enclosing scope, or evaluated in the scope's own body) still holds once a construct's syntax is governed by a _third_ scope that's neither; and whether the violation's own name can be rebound a second time, later in the very same scope, by a construct the rename mechanism has no way to rewrite at all.

## Gaps found and fixed

- **`forbid_vars`'s rename could silently break a closure, producing a `NameError` at runtime.** `_apply_fixes`/`_collect_scope_replacements` renamed a variable only within the exact scope it was assigned in (`iter_within_scope` never crosses into a nested function/lambda/comprehension), but a nested scope reading that variable as a free variable (closure) was left completely untouched. Reproduced: `def outer(response):\n    data = response.json()\n    def inner():\n        return data\n    return inner()` — `--fix` renamed the assignment to `payload` but left `inner`'s `return data` unchanged, so calling the fixed function raised `NameError: name 'data' is not defined` (ch. 2: "MUST NOT perform an auto-fix that can change runtime behavior without the rule explicitly defining that behavior change"; "MUST ensure that a fix does not change name binding or scope unintentionally"). Fixed by extending `_collect_scope_replacements` to recurse into a nested function/lambda/comprehension whenever it doesn't itself rebind the name (new `_binds_name_in_nested_scope`, mirroring `validate_function_name.autofix`'s existing `_binds_name`/`_is_rebound_in_scope` conservative-shadow-check pattern), so the rename follows the variable into every scope that still refers to the same binding. A `global`/`nonlocal` declaration of the name inside a nested scope is treated as "don't recurse" too — those statements store their names as plain strings, not `ast.Name` nodes, so there's no position to safely rewrite there; `atomic_write_text()`'s own `compile()` check (`0010`) is the confirmed backstop if that ever produces an inconsistent `nonlocal`, rejecting the whole fix rather than partially corrupting the file (verified: the file is left byte-identical to the input in that case). A class body is still never recursed into — a pre-existing, unchanged limitation (see below). Regression tests: `test_autofix_follows_closure_reference_into_nested_function`, `..._into_lambda`, `..._into_comprehension`, `test_autofix_does_not_rename_shadowed_reference_in_nested_scope` (parametrized: parameter/local-reassignment/nested-def/nested-import/comprehension-for-target shadowing), `test_autofix_rejects_rather_than_corrupts_nonlocal_reference` in `tests/test_forbid_vars.py`.
- **`misplaced_comment`'s fix hardcoded `"\n"` when rewriting a touched line, silently converting CRLF to LF on exactly the lines it edited.** Both branches of `fix()` (the inline-comment case and the comment-too-long "move above" case) built replacement line text with a literal `\n` instead of reusing the line's own terminator. Reproduced against a CRLF file (`foo(\r\n    bar,\r\n)  # comment\r\n`): after `--fix`, the untouched first line kept `\r\n` but the two rewritten lines ended in bare `\n`, leaving mixed line endings in one file (ch. 3: "MUST preserve the intended newline convention"; ch. 21: "MUST NOT unexpectedly reformat unrelated code", "MUST preserve formatting outside the intended edit range"). Fixed by adding `line_terminator()` (`_base.py`) — returns whichever of `\r\n`/`\n`/`\r` a line ends with, or `""` for a final line with none — and using it in place of every hardcoded `"\n"` in `misplaced_comment.fix()`. As a side effect this also stopped a related, narrower bug: the bracket line's rewrite unconditionally appended `"\n"` even when it was the file's last line with no trailing newline at all, which would have spuriously added one. Regression test: `test_fix_preserves_crlf_on_touched_lines` (parametrized: inline-placement, preceding-placement) in `tests/test_misplaced_comment.py`.
- **`redundant_assignment`'s naive comment-detection heuristic had a false negative that let `--fix` silently delete a real trailing comment.** `_has_inline_comment`/`_has_comment_above` (`analysis.py`) tracked "inside a string" with a single-quote-at-a-time toggle, checking only the immediately preceding character for an escaping backslash. This misparses any line whose string content contains an odd number of the delimiter's own quote character before the comment. Reproduced: `sep = "\\"  # Windows path separator` — the escaped backslash right before the closing quote fooled the one-character lookback into treating that quote as itself escaped, so the scanner never realized the string had closed, and `#` afterward was invisible to it. Since `should_report_violation`'s "skip if there's an inline comment" rule (ch. 2: "MUST preserve comments unless the rule explicitly owns the relevant comment"; ch. 21: "MUST preserve comments where possible") never fired, the assignment was reported and — for a short enough variable name — auto-fixed, blanking the line and deleting the comment along with it. Confirmed end-to-end: `def get_sep() -> str:\n    sep = "\\"  # Windows path separator\n    return sep` → `--fix` → `def get_sep() -> str:\n    return "\\"` (comment gone). Fixed by replacing the text-scan heuristic with `classify_comment_lines()` (new, `_base.py`), which uses `tokenize` — the same lexical grammar Python itself uses, so it can't be fooled by embedded quote parity the way a hand-rolled scanner can — computed once per file in `VariableTracker.__init__` rather than per assignment, to stay `O(source size)` instead of `O(assignments × source size)` (ch. 30: performance). `_has_comment_above`/`_has_inline_comment` were deleted; `AssignmentInfo.has_comment_above`/`has_inline_comment` are now plain set-membership checks against the precomputed classification. Regression tests: `test_classify_comment_lines_trailing_comment` (parametrized, including the escaped-backslash and an embedded-quote-in-triple-quoted-string case), `test_has_comment_above_true_for_standalone_comment` in `tests/redundant_assignment/test_analysis.py`; end-to-end `test_fix_preserves_trailing_comment_on_string_ending_in_escaped_backslash` in `tests/redundant_assignment/test_autofix.py`.
- **Two further gaps were found in the fixes above, both confirmed by direct reproduction before being fixed:**
  - **`forbid_vars`'s new closure-following rename (previous bullet) didn't recognize every kind of scope-shadowing binding, so it could rename _into_ a binding it should have left alone.** `except E as name:`, a match `case name:`/`case {**name}:` capture, and a PEP 695 type parameter (`def f[name]():`) each introduce a new local binding the same way a parameter or assignment does — but each stores its name as a plain string (`ast.ExceptHandler.name`, `ast.MatchAs.name`, `ast.MatchMapping.rest`, `ast.TypeVar.name`/`ast.ParamSpec.name`/`ast.TypeVarTuple.name`), not an `ast.Name` node, so `_binds_name_in_nested_scope`'s original checks (parameters, `Name`+`Store`, def/class name, import, `global`/`nonlocal`) never saw them. Reproduced: `def outer(response):\n    data = response.json()\n    def inner():\n        try:\n            return risky()\n        except RuntimeError as data:\n            return data\n    return inner(), data` → `--fix` → the `except` handler's own `return data` was renamed to `return payload`, silently changing the function from returning the caught exception to returning the unrelated outer JSON payload — and, unlike the `nonlocal` case in the previous bullet, this is _syntactically valid_ Python, so `atomic_write_text()`'s `compile()` check could not catch it (ch. 2: "MUST NOT perform an auto-fix that can change runtime behavior"). Fixed by extending `_binds_name_in_nested_scope` to also treat `ExceptHandler.name`, `MatchAs.name`, `MatchStar.name`, `MatchMapping.rest`, and a `FunctionDef`/`AsyncFunctionDef`'s own `type_params` as shadowing binds. Regression tests: the `except-handler-name`, `match-as-capture`, `match-mapping-rest`, and `type-parameter` cases added to the parametrized `test_autofix_does_not_rename_shadowed_reference_in_nested_scope` in `tests/test_forbid_vars.py`.
  - **The new `classify_comment_lines()` (previous bullet) couldn't see a comment at all on an old-Mac-style CR-only file.** `io.StringIO.readline()` — unlike `ast.parse()`/`split_lines_like_ast()`, which treat a lone `\r` as a line boundary the same as `\n`/`\r\n` — doesn't split on a bare `\r` at all, so `tokenize.generate_tokens()` over such a file saw the entire file as one giant physical line and never emitted a `COMMENT` token on the line that actually had one. Confirmed `ast.parse()` itself accepts bare-`\r` source (so such a file would pass this project's own `check-ast` precondition and reach `ast_checks`) and confirmed the same flaw was already latent in the pre-existing `find_ignored_lines()` (`_base.py`) and both of `misplaced_comment.py`'s own `tokenize` calls — not something newly introduced by `classify_comment_lines()` alone, just newly exercised by it. Fixed by adding `normalize_for_tokenize()` (new, `_base.py`) — replaces a bare `\r` (never one that's part of `\r\n`) with `\n` before tokenizing, a 1-for-1 character substitution that can't change line count/numbering — and calling it at all four `tokenize.generate_tokens()` call sites in the codebase (`find_ignored_lines`, `classify_comment_lines`, and both call sites in `misplaced_comment.py`'s `check()`/`fix()`), per `AGENTS.md`'s "never ignore existing bugs" and "reuse shared code" guidance. Regression tests: `test_normalize_for_tokenize`, `test_classify_comment_lines_on_cr_only_source`, `test_find_ignored_lines_on_cr_only_source` in `tests/test_base.py`; end-to-end `test_check_and_fix_detect_comment_on_cr_only_source` in `tests/test_misplaced_comment.py` (also confirms the fix from the previous bullet's `line_terminator()` correctly preserves the CR-only convention on the touched lines).
- **Three more scope-shadowing constructs were found missing from `_binds_name_in_nested_scope` — all in `forbid_vars`, all confirmed by direct reproduction:**
  - **A walrus (`:=`) target inside a comprehension was wrongly treated as comprehension-local, so it was left unrenamed while a later reference to the same name was renamed — silently splitting one variable into two.** PEP 572 binds a `:=` target inside a comprehension to the nearest _enclosing non-comprehension_ scope, not the comprehension itself — so `_binds_name_in_nested_scope`'s comprehension branch, which reused the same generic "any `Name`+`Store` means shadowed" check the function branch uses, wrongly matched the walrus target's own `Store`-context `Name` node and treated the comprehension as shadowing, refusing to recurse into it at all. Reproduced: `def outer(response, items):\n    data = response.json()\n    return [(data := item) for item in items], data` → `--fix` → `return [(data := item) for item in items], payload` — the walrus still writes to a variable literally named `data` (now unrelated to `payload`), while the trailing reference was renamed and no longer reflects the walrus-updated value: the returned tuple's second element silently reverts from "whatever the comprehension last computed" to "the original `response.json()` value" (ch. 2: "MUST NOT perform an auto-fix that can change runtime behavior"). Fixed by restructuring `_binds_name_in_nested_scope` so the comprehension branch only ever checks `for` targets (the _only_ thing that can locally bind a name directly within a comprehension's expression-only grammar) and never falls through to the generic `Store`-based check at all. Regression test: `test_autofix_renames_walrus_target_that_escapes_comprehension_to_outer_scope` in `tests/test_forbid_vars.py`.
  - **`del name` inside a nested scope wasn't recognized as a binding, so a later local reassignment of the same name in that scope was wrongly renamed as if it were the outer variable.** Python's "any binding operation anywhere in a scope makes the name local to the whole scope" rule applies to `del` exactly as it does to `=`, but a `del` target's `ast.Name` node has `ctx=ast.Del()`, not `ast.Store()`, so the existing `isinstance(child.ctx, ast.Store)` check missed it. Reproduced: a nested function with `del data` followed by its own, unrelated `data = "local value"; return data` had that local reassignment and read renamed to `payload`, even though they have nothing to do with the outer `payload = response.json()`. Fixed by widening the check to `isinstance(child.ctx, ast.Store | ast.Del)`. Regression test: the `nested-del-then-reassignment` case added to `test_autofix_does_not_rename_shadowed_reference_in_nested_scope`.
  - **A dotted `import a.b.c` (no `as`) wasn't recognized as binding `a`, so a nested scope's own attribute access through that import was wrongly rewritten.** `ast.alias.name` for a dotted import is the full path (`"a.b.c"`), but Python only binds the first component (`"a"`) in the local namespace — the existing check compared `alias.name` directly against the candidate name, which never matches a multi-component path. Reproduced: a nested function with `import data.models; return data.models` had the attribute access rewritten to `return payload.models` — a reference to a nonexistent attribute on an unrelated value, since `payload` was never a module. Fixed by splitting `ast.Import`'s own check out from `ast.ImportFrom`'s (whose aliases never contain a dot in the first place) and comparing against `alias.name.split(".")[0]`. Regression tests: the `dotted-import` and `from-import` cases added to `test_autofix_does_not_rename_shadowed_reference_in_nested_scope`.
- **A shadowed scope's own signature could itself reference the outer variable — treating a shadowed nested scope as fully opaque skipped parts of it that actually run in the _enclosing_ scope, reintroducing the exact `NameError` class the original fix set out to prevent.** Reproduced two ways: `def outer(response): data = response.json(); return [data for data in data]` → `--fix` → `return [data for data in data]` (only the assignment renamed) → calling `outer()` raised `NameError: name 'data' is not defined`, because a comprehension's _first_ `for` clause's iterable is evaluated in the enclosing scope before the comprehension's own for-target starts shadowing anything — but `_binds_name_in_nested_scope` returning `True` for the comprehension's own for-target caused the _entire_ comprehension, iterable included, to be skipped. Likewise `def outer(response): data = response.json(); def inner(data=data): return data; return inner()` → `--fix` → `def inner(data=data): return data` (again, only the assignment renamed) → calling `outer()` raised the same `NameError`, since a parameter default is evaluated at `def`-time in the enclosing scope, not the function's own body scope, yet the parameter's own name shadowing the function's body caused the whole `FunctionDef` — defaults included — to be skipped (ch. 2: "MUST NOT perform an auto-fix that can change runtime behavior"). A related but distinct bug surfaced while fixing this: the initial fix (visiting each `_outer_scope_children()`/`_own_scope_children()` node through the existing children-only `_collect_scope_replacements()`) missed a bare-`Name` outer-scope child entirely (a default value or comprehension element that's _just_ `data`, not a container holding it), and separately failed to apply a nested function's _own_ shadow-check when that function was reached as someone else's body statement rather than as `_collect_scope_replacements`'s own top-level `scope` argument (both silently reproduced the corresponding bug again). Fixed by: (1) adding `_outer_scope_children()`/`_own_scope_children()` (new), which split a `FunctionDef`/`AsyncFunctionDef`/`Lambda`/comprehension's own fields into the enclosing-scope-evaluated ones (decorators, defaults, annotations — except when PEP 695 type parameters are present, confirmed against CPython that annotations then move into the type parameters' own implicit scope while defaults still don't — and a comprehension's first iterable) versus the scope's own (body, or every generator past the first); (2) replacing `_collect_scope_replacements()`'s internals with a single unified recursive `_collect_replacements()` that checks every node — including the node passed in, not just its children — and applies the shadow-check/outer-own split to _any_ crossable scope it encounters, regardless of how it was reached; `_collect_scope_replacements()` itself is now a thin entry point preserving the original "only inspect `scope`'s children, `scope` itself is never shadow-checked" contract the top-level caller in `_apply_fixes` needs. Regression tests: the parametrized `test_autofix_renames_reference_evaluated_in_enclosing_scope` in `tests/test_forbid_vars.py` (comprehension-first-iterable, comprehension-later-iterable, dict-comprehension, function-default-and-annotations, vararg-kwarg-annotations, type-params-annotation-not-shadowed, lambda-default, type-params-no-return-annotation).
- **Three more gaps were found: two further ways `forbid_vars`'s rename could silently produce a wrong result without any crash to signal it, and one more comment-classification gap in `classify_comment_lines()` (independent of `forbid_vars`), all confirmed by direct reproduction:**
  - **Two independent violations in different but nested (non-shadowing) scopes could be given the same suggested replacement name, colliding once the outer one's rename was followed into the inner scope.** `_generate_unique_name`'s uniqueness check only considered the current scope's own already-used suggestions, unaware that closure-following (the closure-following fix described above) could land an ancestor scope's rename inside this one too. Reproduced: `def outer(response): data = response.json(); def inner(response2): result = response2.json(); return data, result` — both `data` and the inner `result` match the same `.json()` autofix pattern ("payload") — `--fix` → `return payload, payload`, silently making both returned values identical (ch. 2: "MUST NOT perform an auto-fix that can change runtime behavior"). Fixed two ways: (1) `_get_scope_names()` now walks a scope's _entire_ subtree (crossing nested scopes) instead of just its own immediate names, so a suggestion also avoids an existing identifier that merely lives in a nested scope; (2) new `_ancestor_used_suggestions()` unions the current scope's own used-suggestions bucket with every _ancestor_ scope's bucket, so a nested violation avoids a name an enclosing scope's own violation already claimed — `inner`'s `result` now gets `payload_2` instead of colliding. This guards the common ordering (variable assigned, then a closure over it defined afterward); a closure defined _before_ the variable it will eventually capture is assigned, with an independently-matching suggestion, is a rarer, undocumented residual gap the ancestor-only check doesn't close (would need a full two-pass suggestion-generation redesign, disproportionate for a single-maintainer hobby project's default two-name list). Regression tests: `test_autofix_avoids_cross_scope_suggestion_collision`, `test_autofix_avoids_suggestion_colliding_with_existing_nested_name` in `tests/test_forbid_vars.py`.
  - **A module-level (or enclosing-function-level) variable read or written via `global`/`nonlocal` inside a nested function was treated as shadowed and skipped entirely — but a `global`/`nonlocal` declaration doesn't introduce a new binding at all, it explicitly refers to the very variable being renamed.** The closure-following fix described above already refused to rename _inside_ such a nested scope (correctly, since the declaration's own name is a plain string with no rewritable position) but reproduced worse than a `NameError` for the `global`-read case, and a second, more severe failure mode for the `global`-write case: `data = None\n\ndef loader(response):\n    global data\n    data = response.json()\n\ndef reader():\n    global data\n    return data` — `--fix` renamed only the module-level `data = None` to `payload = None`, leaving `loader`'s `global data; data = response.json()` and `reader`'s `global data; return data` completely untouched. Since `def loader(): global data; data = response.json()` (unrenamed) is still perfectly valid syntax that just writes to a _different_, now-stale module attribute called `data` instead of the renamed `payload`, `atomic_write_text()`'s `compile()` check has nothing to reject — the caching pattern this represents (a module-level value populated by one function, read by another) would silently stop updating the renamed variable at all, with no error anywhere (ch. 2: "MUST NOT perform an auto-fix that can change runtime behavior"). Properly _following_ such a reference would require rewriting text inside the `global`/`nonlocal` statement itself — a capability this codebase's purely `ast.Name`-position-based rename mechanism doesn't have — so instead of attempting that, `check()` now refuses to suggest a fix at all for any name mentioned in a `global`/`nonlocal` statement anywhere within its own enclosing scope's subtree (new `ForbiddenNameVisitor._referenced_via_global_or_nonlocal`), keeping `fixable` honest rather than offering a fix that can silently do the wrong thing. This also supersedes the earlier `nonlocal`-mutation case (previously caught only after the fact, by `atomic_write_text()`'s rejection) with a strictly better outcome: the violation is never offered as fixable in the first place, so `_binds_name_in_nested_scope`'s own, now-provably-unreachable `Global`/`Nonlocal` check was deleted rather than left as dead code. Regression tests: `test_autofix_never_offered_for_module_global_read_in_function` (new), `test_autofix_never_offered_for_name_referenced_via_nonlocal` (renamed/rewritten from the earlier `..._rejects_rather_than_corrupts_nonlocal_reference`, which no longer applies now that the violation isn't offered as fixable at all) in `tests/test_forbid_vars.py`.
  - **`classify_comment_lines()` still missed a real trailing comment when it followed a multiline string, independent of any `forbid_vars` scope-tracking issue.** `tokenize` reports a multiline token's (e.g. a triple-quoted string's) position using only its _start_ line — the code-line-tracking loop recorded only that start line as "code," so a comment trailing the string's _closing_ line (a later line) was misclassified as comment-only rather than trailing, since that closing line was never marked as containing code at all. Reproduced: `x = """\nmulti\nline\n"""  # trailing comment\ny = 5` — `y = 5`'s `has_comment_above` check incorrectly returned `True` (treating the closing-quote-plus-comment line as a preceding standalone comment), silently suppressing an otherwise-legitimate `redundant-assignment` report for `y` (ch. 2/21: comment detection must reflect the actual parsed source, not an under-counted token span — lower severity than the two `forbid_vars` findings above, since under-reporting a violation doesn't corrupt anything, but still a real correctness gap in code introduced by this same audit). Fixed by recording every line in a non-comment token's full `start`-to-`end` span as code, not just its start line. Regression tests: `test_classify_comment_lines_multiline_string_closing_line_is_code` in `tests/test_base.py`; end-to-end `test_check_reports_assignment_after_multiline_string_with_trailing_comment` in `tests/redundant_assignment/test_autofix.py`.
- **Two more ways `forbid_vars`'s rename could silently collide with or corrupt an unrelated binding were found, plus an independent reproduction of the reverse-ordering suggestion-collision gap described above, all confirmed by direct reproduction:**
  - **A nested scope's own parameter, never itself an `ast.Name` node, wasn't checked when picking a suggested replacement name, so a rename could collide with a closure's own parameter and silently rebind the closure read to it instead of the renamed outer variable.** `_get_scope_names()` only collected `ast.Name` ids — a parameter name lives on `ast.arg.arg`, not an `ast.Name`, so it was invisible to the uniqueness check `_generate_unique_name()` relies on. Reproduced: `def outer(response): data = response.json(); def inner(payload): return data; return inner(5)` — `--fix` suggested `payload` for `data` (the default `.json()` → "payload" mapping) without noticing `inner` already has its own parameter called `payload`, producing `def inner(payload): return payload` — a syntactically valid rename that now reads `inner`'s own argument instead of the closure, silently discarding the outer JSON value (ch. 2: "MUST NOT perform an auto-fix that can change runtime behavior"). Fixed by widening `_get_scope_names()` to collect every kind of Python binding construct's name — `ast.arg`, `FunctionDef`/`AsyncFunctionDef`/`ClassDef`, `ExceptHandler`/`MatchAs`/`MatchStar`, `MatchMapping.rest`, `TypeVar`/`ParamSpec`/`TypeVarTuple`, `alias` — not just `ast.Name`, the same construct list `_binds_name_in_nested_scope` already treats as shadowing. Regression test: `test_autofix_avoids_suggestion_colliding_with_nested_parameter_name` in `tests/test_forbid_vars.py`.
  - **A walrus target inside a nested function's own default value was wrongly treated as a body-level shadow, so the default got renamed while the body's closure read was left stale, splitting one variable into two.** `_binds_name_in_nested_scope`'s generic scan walked `scope_node` wholesale (`iter_within_scope(scope_node)`), which includes `_outer_scope_children()` positions (decorators, defaults, annotations without type parameters) even though those actually run in the _enclosing_ scope at `def`-time, not the function's own body scope. Reproduced: `def outer(response): data = response.json(); def inner(x=(data := response.json())): return data, x; return inner()` — `--fix` renamed the default's walrus target to `payload := response.json()` (correctly, since it evaluates in `outer`'s scope) but left the body's `return data, x` untouched, because the walrus's `Store`-context `Name` node was wrongly read as `inner` binding `data` in its own scope — reintroducing the exact `NameError` class the original closure-following fix set out to prevent. Fixed by adding `_iter_own_scope_descendants()` (new), which yields only a scope's `_own_scope_children()` and their own descendants, and using it in place of `iter_within_scope(scope_node)` in `_binds_name_in_nested_scope`'s generic scan. Regression test: `test_autofix_renames_walrus_target_inside_default_evaluated_in_enclosing_scope` in `tests/test_forbid_vars.py`.

- **One further collision the reserved-name set still missed was found, plus a quadratic-traversal performance gap in the check introduced by the `global`/`nonlocal` handling described above — one correctness fix and one performance fix, both in `forbid_vars`, both confirmed by direct reproduction:**

Decorators are real AST nodes (`decorator_list`) no fixer strips or reorders.
- **Ch. 2/21, `forbid_vars` class-body closure references**: `_collect_scope_replacements` still never recurses into a `ClassDef` body. A bare (non-`self.`/`cls.`) reference to an enclosing scope's variable directly in a class body (e.g. a locally-defined class's attribute default reading a function-local variable) is a real but rare pattern this audit's fix doesn't close — pre-existing behavior, unchanged by this audit, and consistent with `ForbiddenNameVisitor.visit_ClassDef` already excluding class-level attribute assignments from detection entirely for the same reason (the class name already provides context, per that method's own docstring). Not fixed: correctly distinguishing "a class body's own execution-time scope still sees its enclosing function" from "a method body does not" is one of Python's trickiest scoping quirks, and the existing detection-side exclusion already means this gap can only be reached via a variable _reference_ the class body's own attribute default expression makes, not a rename source — disproportionate to chase down for a hobby project's default `data`/`result` name list.
- **Ch. 2, encoding declarations / BOM**: identical requirement to ch. 3, already audited in depth by `0011` (`atomic_write_text()` always round-trips the `encoding` a file was read with; no fixer's edit logic ever touches a PEP 263 cookie line's content, since fixers only ever edit lines their own AST-derived position lands on).
