# Investigation report: behavioral contract audit, testing requirements & auto-fix pipeline testing (ch. 31, 32)

Detailed findings behind `docs/adr/0026-behavioral-contract-audit-testing-requirements.md`. `docs/behavioral_contract.md` chapters **31** (Testing Requirements) and **32** (Testing the Auto-Fix Pipeline) were checked line-by-line against the entire test suite (`tests/`), following on from `0011`'s fix-engine audit, `0023`'s semantic/formatting-preservation audit, and `0025`'s determinism/idempotence audit, per issue #44 (blocked by, and sequenced after, #30/#39/#40, so this audit targets the corrected behavior those tickets left behind rather than a pre-fix baseline).

Method: every MUST/MUST NOT bullet in both chapters was mapped to the specific test(s) that cover it, by name. Four genuine gaps were found; everything else was already satisfied, in most cases by test infrastructure `0001`â€“`0025` already built while fixing the correctness issues those audits found.

## Gaps found and fixed

- **Ch. 31, "MUST test large files"**: no test anywhere in the suite ran the pipeline against a large file. The only prior evidence was a manual, uncommitted benchmark (`docs/audits/0004`, a 2000-function/120KB file, run ad hoc during that investigation to measure a since-fixed quadratic `get_source_segment` call). Added `test_process_files_handles_a_large_file` (`tests/test_orchestrator.py`): generates a 300-function file, runs `forbid-vars` in fix mode over it, and asserts every violation is found and fixed, nothing lands in `unprocessable_files`/`rule_failures`, and the forbidden pattern is gone from the final file. Deliberately a correctness check, not a timing assertion â€” ch. 24/30's own performance requirement is `0004`'s territory, already covered by that fix's own unit-level regression tests (`test_fast_get_source_segment_matches_ast_get_source_segment`).
- **Ch. 31, "MUST test paths containing spaces and Unicode"**: no test used such a path, even though ch. 13's underlying requirement was already satisfied by design (`0006`/`0015`'s file-discovery audit: every path is a plain subprocess/filesystem argument, never passed through a shell) â€” that audit's own report only records a manual, ad hoc verification ("verified empirically with a path literally containing `;&&|$(...)`"), never committed as a test. Added `test_main_handles_path_containing_spaces_and_unicode` (`tests/test_orchestrator.py`): a real `main()` invocation against a file under a directory named `my proj cafĂ© ć—Ąćś¬čŞž`, asserting discovery, checking, fixing, and the final file content all work correctly. Three refinements during review, each confirmed by deliberately reintroducing the regression it targets and watching the test fail, then reverting:
  - The first version ran outside any git repo (this file's usual `tmp_path` convention) â€” `forbid-vars` declares a prefilter pattern, so that made `git_grep_filter()`'s real `git grep` subprocess call error out on a path outside any repository and silently fall back to the Python-only path, which has no subprocess argument-passing to prove safe in the first place. Fixed by running the test inside a real git repo (`git init`/`git add`, matching `test_git_grep_filter_real_success_and_no_match_paths`'s own setup in `tests/test_prefilter.py`).
  - Even inside a real repo, the test still only asserted `main()`'s end-to-end result â€” which a silent fallback to the Python path reproduces identically, so it still wouldn't have caught a regression in the actual git-grep subprocess call. Confirmed by deliberately mangling the path-construction code that feeds `git grep`'s own argument list (stripping spaces before the `subprocess.run` call): every original assertion (`exit_code`, `[FIXED]`, final file content) still passed, because the Python fallback silently produced the same correct result. Fixed by spying on the real `subprocess.run` (still executed for real, never mocked out) to record every `git grep` invocation's `returncode`/`stderr`, asserting at least one succeeded (`returncode == 0`) with empty `stderr` on every call.
  - Recording only the result, not the command itself, still missed one case: a regression that dropped the path argument from the `git grep` command entirely would search the whole (single-file) test repository, still find the sole file, and pass every assertion above despite silently losing path scoping. Confirmed by deliberately dropping the pathspec from the command list: the previous assertions still passed. Fixed by also recording each call's full command list and asserting the exact path string appears in every one of them.
- **Ch. 32, "MUST verify that the reported result matches the actual final file state"**: `test_main_fix_flag_marks_violation_fixed` â€” the one test exercising a _successful_ `--fix` run through `main()`'s own printed output â€” only asserted the `[FIXED]` line appeared, never the file's actual resulting content. Its three sibling tests (`test_main_fix_flag_reports_rejected_fix`, `..._errored_fix`, `..._failed_fix`) already assert the file stays byte-for-byte untouched; the success path was the one outcome this exact requirement names that had no matching assertion. Extended the test with `assert filepath.read_text() == "response = requests.get(url)\nprint(response)\n"`.
- **Ch. 31, "MUST NOT consider a test that merely checks that the process did not crash sufficient evidence of correctness"**: `test_install_sigterm_handler_degrades_when_signal_signal_unavailable` was a direct instance of the pattern this line prohibits â€” it monkeypatched `signal.signal` to always raise, called `_install_sigterm_handler()`, and asserted nothing except that the call itself didn't raise. Strengthened it to assert the actual degraded end state: `signal.getsignal(signal.SIGTERM)` is unchanged from before the call (no partial/broken handler was installed), and the fallback path's `logger.debug("Could not install a SIGTERM handler; continuing without one.")` was actually reached (via `caplog`).

## Already satisfied, confirmed with citations

Ch. 31 items, in the chapter's own order:

- **Valid input / invalid or unsupported input**: every one of the six checks has both `good`/`bad` (and several `ignore`) fixtures under `tests/fixtures/`, plus dozens of unit tests feeding invalid/edge-case input directly to `check()`/`fix()` â€” e.g. `redundant_assignment`'s `test_autofix_skips_violation_with_invalid_fix_data`, `test_autofix_skips_invalid_line_indices`, `test_autofix_with_invalid_assignment_line`; `validate_function_name`'s `unsafe_large.py`/`unsafe_complex.py` autofix-refusal fixtures.
- **Syntactically valid Python**: enforced structurally by `atomic_write_text()` (every check's only write path) and directly tested by `test_atomic_write_text_rejects_invalid_syntax`; every other fixture-based fix test in the suite transitively re-proves this, since a fix producing invalid syntax would raise `FixValidationError` and fail the test rather than silently pass.
- **Preserves intended semantics**: the subject of `0023`'s entire audit â€” 30+ regression tests in `tests/test_forbid_vars.py` alone, cited in full in `docs/audits/0001`.
- **Fix idempotence**: `test_fix_converges_after_one_pass_across_all_checks` (`0025`), parametrized over all 33 `tests/fixtures/**/bad/*.py` fixtures.
- **Conflicting fixes / overlapping fixes / multiple fixes in the same file**: `test_apply_fixes_recomputes_stale_positions` (two checks, one's fix shifting the lines the other must then fix correctly), `test_apply_fixes_refreshes_non_fixable_checks_position_too`, `test_apply_fixes_refreshes_a_participating_checks_own_left_open_violation`, and `test_apply_fixes_marks_only_the_rejected_violation_of_a_multi_write_check`/`..._errored_violation...` (multiple violations from one check in one file, one resolved and one not). No two of this project's 6 checks are designed to transform the same construct in opposite directions (`0025`'s ch. 10 finding, re-confirmed here), so a same-span "conflict" between two different checks' fixes isn't a constructible scenario to test beyond the position-recompute behavior these tests already cover.
- **Files with comments**: the entire `misplaced_comment` suite plus `classify_comment_lines`'s own parametrized tests in `tests/test_base.py`.
- **Files with Unicode**: `test_check_reports_character_offset_not_byte_offset_before_multibyte_text` (three checks), `test_autofix_replaces_name_on_line_with_non_ascii_text`, `test_fix_inlines_use_on_line_with_non_ascii_text`, `test_apply_fix_renames_call_site_on_line_with_non_ascii_text`.
- **Different newline conventions**: `test_fix_preserves_crlf_on_touched_lines`, `test_check_and_fix_detect_comment_on_cr_only_source`, `test_fix_preserves_crlf_line_endings`, `test_classify_comment_lines_on_cr_only_source`, `test_find_ignored_lines_on_cr_only_source`.
- **Encoding declarations**: `test_fix_honors_pep263_encoding_declaration`; the `bad-encoding-cookie`/`undecodable-content` cases of `test_process_files_unreadable_file_is_skipped`.
- **Empty files**: `excessive_blank_lines`'s `empty-file` fixture case and `test_fix_file_content_empty_source_returns_unchanged`.
- **Very small files**: `test_main_no_violations_returns_zero` (a one-line file through `main()`) and the overwhelming majority of this suite's unit tests, which use 1-3 line sources.
- **Malformed Python**: the `invalid-syntax` case of `test_process_files_unreadable_file_is_skipped`, `test_main_unparseable_file_returns_one`, `tests/fixtures/invalid_code.py`. No check itself is tested against malformed Python directly (each receives an already-parsed `ast.AST`) â€” consistent with `0011`'s finding that this project's own precondition (`AGENTS.md`: every file already passed `check-ast`/ruff) makes that this codebase's only parsing boundary, already covered above.
- **Missing files**: the `missing-file` case of `test_process_files_unreadable_file_is_skipped`, `test_git_grep_filter_includes_file_deleted_since_discovery` (real `git` subprocess), `test_expand_directories_skips_tracked_file_deleted_from_working_tree`.

- **Complete pipeline, not just edit-generation functions**: `tests/test_orchestrator.py`'s `test_main_*` tests already drive `main()` end to end â€” CLI parsing, discovery, reading, parsing, checking, fixing, writing, reporting â€” for the large majority of this suite's pipeline-level tests, not just individual `check()`/`fix()` unit calls.
- **The 10 listed pipeline stages**, each independently confirmed to have failure-mode coverage:
  1. _source discovery_: `test_expand_directories_*`, `test_main_directory_argument_checks_files_inside_it`, `test_expand_directories_falls_back_when_git_ls_files_fails`.
  2. _reading_: all four cases of `test_process_files_unreadable_file_is_skipped`.
  3. _parsing_: the `invalid-syntax` case above, `test_main_unparseable_file_returns_one`.

  7. _source reconstruction_: every check's own `test_fix_write_failure_returns_false`.

- **Failure at every significant stage**: follows from the same 10-stage mapping above â€” every stage has at least one dedicated failure-path test.
- **Failures do not corrupt the source file**: `test_main_fix_flag_reports_rejected_fix`/`..._errored_fix`/`..._failed_fix` each assert the file is byte-for-byte unchanged; `test_real_sigterm_mid_run_stops_gracefully_without_leftover_temp_files` asserts every file ends up in one of exactly two valid states (untouched or fully fixed), never partial.
- **Interruption at safe and unsafe points**: three distinct interruption points are covered â€” before any write begins (`test_atomic_write_text_rejects_invalid_syntax`, rejected before the temp file even exists), between the temp-file write and the rename (`test_atomic_write_text`'s directory-in-place case), and at file granularity between separate checks/files within one run (`test_real_sigterm_mid_run_stops_gracefully_without_leftover_temp_files`, a real `os.kill(..., SIGTERM)` mid-batch).

## Items judged not applicable, with rationale

- **Ch. 31, "MUST test serial and parallel execution"**: `0007`'s audit already confirmed, by grep, that no threading or multiprocessing exists anywhere in this codebase's own execution â€” every check runs sequentially, always. There is no in-process parallel mode to test as a separate condition from the serial one already tested throughout this suite. The one real form of concurrency this project has â€” multiple external `prek`-dispatched processes racing over the same cache â€” is a cross-process concern already tested by `test_concurrent_writers_do_not_lose_updates`.
- **Ch. 31, "MUST test all officially supported operating systems"**: Linux (or WSL) is this project's only officially supported platform (`AGENTS.md`: "Develop and test against Linux (or WSL) only"; `docs/adr/0020`). "All officially supported operating systems" is a one-item list, and every test in this suite already runs on it â€” there is no second platform to add coverage for without first expanding the project's own support scope, which is a product decision, not a testing gap.

## Consequences

- `tests/test_orchestrator.py` gains `test_main_handles_path_containing_spaces_and_unicode` and `test_process_files_handles_a_large_file`; `test_main_fix_flag_marks_violation_fixed` gains a final-file-content assertion.
- `tests/test_main.py`'s `test_install_sigterm_handler_degrades_when_signal_signal_unavailable` now asserts the actual degraded end state (handler left uninstalled, fallback logged) instead of only the absence of an exception.
- No production code changed; every finding here was a test-coverage gap, not a behavior gap.
