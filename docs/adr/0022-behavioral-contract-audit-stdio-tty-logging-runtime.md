# Behavioral contract audit: stdin/stdout/TTY, logging & debugging, installation and runtime environment (ch. 26, 27, 29)

`docs/behavioral_contract.md` chapters **26** (Standard Input, Output, and TTY Behavior), **27** (Logging and Debugging), and **29** (Installation and Runtime Environment) were checked line-by-line against `_cli.py`'s argument parsing and `main()`, `__main__.py`'s process entry point, `_diagnostics.py`'s report formatting, every module's own `logging.getLogger(...)` usage, `_cache.py`'s `fcntl` handling, `_discovery.py`'s/`_prefilter.py`'s `git` subprocess fallbacks, and `pyproject.toml`'s dependency/version metadata, following on from `0011`–`0021`'s prior audits, per issue #38.

Grepped the whole source tree for every pattern each chapter could plausibly touch: `stdin`, `isatty`, `tty`/`TTY`, `input(`, `setpgrp`/`tcgetpgrp`/`tcsetpgrp`, ANSI/color escape codes, `basicConfig`/`setLevel`/`addHandler`/`isEnabledFor`, and `os.environ`/`getenv` — every one of those came back empty except the logging-related search, which found every check's own `logger = logging.getLogger(...)` and its `logger.debug`/`logger.warning`/`logger.exception` call sites. `pyproject.toml`'s `[project]` table has no `dependencies` key at all — this project has zero runtime dependencies beyond the standard library — and `0021` already established there's no plugin mechanism (`ALL_CHECKS` is a fixed, hand-authored list).

## Already satisfied, confirmed with citations

- **Ch. 26, every item**: this tool never reads from, writes to, or queries a terminal in any TTY-aware way — no `stdin` read path exists at all (`filenames` are file paths only; a literal `-` argument is treated as an ordinary, nonexistent file and cleanly reported via the existing `"{filepath}: error: could not be read or parsed; file skipped"` path, verified with a live `uv run ... - < /dev/null` invocation: exit 1, no hang), no `isatty()`/`tty`/`termios`/`curses` call and no ANSI/color escape code anywhere (grepped, confirmed empty), and no `input()` prompt or foreground-process-group assumption (`setpgrp`/`tcgetpgrp`, also grepped empty). Every diagnostic is a plain `print(..., file=sys.stderr)` line (`_diagnostics.py`), identical regardless of whether stdout/stderr are a TTY, a pipe, a redirected file, or a CI log collector — verified live with `< /dev/null` stdin and piped stdout/stderr (no hang, correct exit code). "MUST work correctly when invoked through a process supervisor" and "through pipes"/"from CI" additionally rest on the cancellation-safety work `0016` already audited (SIGTERM handling, no orphaned `git`/`git grep` children) and the 30s subprocess timeouts `0013` already audited — nothing here is TTY-specific, so there's nothing left for chapter 26 to add on top of "no terminal-dependent code path exists."
- **Ch. 27, "MUST make it possible to identify the relevant file, rule, and operation during internal failures"**: already true without any log-level change — `_diagnostics.py`'s normal-output report names the exact file and `check_id` for both an unprocessable file and a rule crash (`orchestrator.unprocessable_files`/`rule_failures`, both audited in depth by `0012`).
- **Ch. 27, "MUST NOT log secrets or sensitive source content unnecessarily"; "MUST avoid logging enormous source contents by default"**: grepped every `logger.*(...)` call site in `src/` — every one logs only a `filepath`, a `check_id`, or `repr(error)`; none references a `source`/`content` variable holding actual file text.
- **Ch. 27, "MUST ensure that debug logging does not change lint results"; "does not change auto-fix behavior"**: no call site anywhere branches on `logger.isEnabledFor()`/`getEffectiveLevel()` (grepped, confirmed empty) — every `logger.debug(...)` call is a pure side effect with no return value consulted by check/fix logic, so raising the log level cannot change what is reported or fixed by construction, not just by testing.
- **Ch. 27, "MUST ensure that log output does not corrupt structured output modes"**: moot — there is no structured (e.g. JSON) output mode to corrupt; the only output format is the plain-text stderr report `_diagnostics.py` already produces.
- **Ch. 29, "MUST provide a clear error when required runtime dependencies are unavailable"; "MUST avoid silently using an incompatible dependency version"**: there are no required runtime dependencies to be unavailable or mismatched — `pyproject.toml`'s `[project]` table has no `dependencies` key, and every import across `src/` (grepped and enumerated) is either always-present stdlib (`argparse`, `ast`, `json`, `os`, `pathlib`, `re`, `subprocess`, `sys`, `tempfile`, `tokenize`, ...) or `fcntl`, the one POSIX-only stdlib module, already handled below.
- **Ch. 29, "MUST handle missing optional dependencies gracefully"; "MUST clearly distinguish optional-feature failure from core linter failure"**: already fully satisfied by two pre-existing, independent fallbacks, one per optional acceleration this codebase has. `_cache.py`'s `import fcntl` (`0020`, cross-platform-behavior audit) degrades to a disabled-but-correct cache with one explicit `logger.warning(...)` when `fcntl` is unavailable — a warning specifically because the fallback changes observable behavior (uncached instead of cached) and that difference is worth surfacing. `_discovery.py`'s/`_prefilter.py`'s `git ls-files`/`git grep` calls fall back to an equivalent pure-Python `rglob`/substring scan on any `subprocess.SubprocessError`/`FileNotFoundError`/`TimeoutExpired`, logged only at `DEBUG` — deliberately silent by default because that fallback produces the identical candidate set, just slower, so nothing has actually gone wrong from the user's perspective (both fallbacks' own code comments state this distinction explicitly). Between them, every optional dependency this codebase has is already both gracefully degraded and correctly distinguished by severity.
- **Ch. 29, "MUST support the officially supported Python runtime versions"; "MUST ensure that installed plugins are compatible with the core linter version"; "MUST define behavior when multiple versions of a plugin or dependency are present"**: the first is `0021`'s own citation (`requires-python = ">=3.14"`, `AGENTS.md`); the latter two don't apply — no plugin mechanism exists (`0021`, ch. 18).

## Gap found and fixed

- **Chapter 27's "MUST provide a debug or verbose mode when normal output is insufficient for troubleshooting" had no way to reach it through this tool's own CLI.** Every module already logs the real exception behind a reported failure via `logger.debug(..., exc_info=True)` — deliberately, per each call site's own comment, so it doesn't duplicate the clean diagnostic line `_diagnostics.py` already prints for the same failure (`0017`'s "several internal `logger.exception()` calls leaked a full raw traceback" fix downgraded these to `DEBUG` specifically to fix that duplication) — but nothing in this codebase ever calls `logging.basicConfig()`, `setLevel()`, or reads an env var to raise the level, and no CLI flag existed to do so either. A user who hit `"check 'X' raised an unexpected exception; its results for this file may be incomplete"` (or the equivalent unreadable-file/cache-serialization message) had no supported way to see _why_ — not via `.pre-commit-hooks.yaml`'s plain `entry: python -m pre_commit_hooks.ast_checks` invocation, and not via any documented flag or environment variable. The only way to see the already-captured detail was to bypass the CLI entirely and write custom Python that imports this package and calls `logging.basicConfig()` itself before invoking `main()` — not a real troubleshooting path for a pre-commit/prek hook run as a subprocess. Fixed by adding `-v`/`--verbose` to `_cli.py`'s argument parser (mirroring `ruff`'s own `-v`/`--verbose` flag, consistent with this project's existing `ruff check` CLI-parity design, `0008`): when set, `main()` calls `logging.basicConfig(level=logging.DEBUG, stream=sys.stderr)` once, immediately after parsing args and before any file processing, surfacing every already-existing debug-level log line (and its `exc_info` traceback) for the rest of the run. No other behavior changes: the flag only reconfigures logging, so it cannot affect which violations are reported or how `--fix` behaves (ch. 27's own explicit "MUST NOT" — verified, not just asserted, by `test_verbose_flag_does_not_change_violations_or_exit_code`, which diffs a `--verbose` and a non-`--verbose` subprocess run of the same violation byte-for-byte on the parts that matter: exit code and the violation line itself). Regression tests: `test_verbose_flag_surfaces_the_underlying_exception_on_stderr` and `test_verbose_flag_does_not_change_violations_or_exit_code` (`tests/test_main.py`), both real subprocess invocations — matching `test_real_invocation_does_not_leak_a_traceback_onto_stderr`'s own established pattern in the same file, since pytest's own logging-capture plugin attaches a handler to the root logger for the whole test session, masking `logging.basicConfig()`'s real no-configuration-yet default from any in-process call. `test_main_verbose_flag_does_not_change_a_clean_runs_exit_code` (`tests/test_orchestrator.py`) additionally exercises the new branch in-process, needed only for coverage — the two subprocess tests above are the actual behavioral proof.

## Items judged not applicable, with rationale

- **Ch. 29, "MUST report incompatible runtime versions clearly"**: not implemented as an in-code runtime check, deliberately. This project's own source already uses syntax that doesn't exist before Python 3.14 (PEP 758 `except A, B:`, `0021`) in its own entry-point module (`__main__.py`), so an incompatible interpreter fails to even _compile_ `__main__.py` — no runtime check written in this codebase could run early enough to intercept that, since the whole module must parse successfully before any of its own code executes, regardless of where in the file a version check appears. The only way to place such a check ahead of the failure would be a second, separate shim module deliberately written in a syntax subset compatible with much older Pythons, whose only job is a version gate before delegating to the real entry point — exactly the kind of "compatibility branch" `AGENTS.md` explicitly forbids adding ("must not bloat the code with compatibility branches — this limits ongoing maintenance effort"). In practice this scenario is already caught one layer up, before this project's own code ever runs: `pip`/`uv` both read `requires-python = ">=3.14"` during install and refuse with their own clear, standard error (`ERROR: Package 'ruff-extra-rules' requires a different Python: 3.11.x not in '>=3.14'`) — the same mechanism the vast majority of Python packages rely on rather than re-implementing their own version gate. Reaching this project's own code under an incompatible interpreter at all requires bypassing normal installation (e.g. hand-copying `src/` onto an unrelated interpreter's `sys.path`), a self-inflicted, unsupported scenario outside what `AGENTS.md`'s Python-version policy commits to.

## Consequences

- `_cli.py` gains a `-v`/`--verbose` flag; `main()` calls `logging.basicConfig(level=logging.DEBUG, stream=sys.stderr)` once when set, before any file processing. No change to argument parsing for existing flags, no change to `check()`/`fix()` behavior, no change to the exit-code contract (`0012`).
- `tests/test_main.py` gains two subprocess-based regression tests; `tests/test_orchestrator.py` gains one in-process coverage test for the same branch.
- No other production code changed — chapters 26 and 29 were already fully satisfied by this codebase's existing design (no TTY-dependent code path at all; the two optional-dependency fallbacks `0020` and the pre-existing `git` fallback already handle chapter 29's "optional dependency" items correctly).
