# Behavioral contract audit: caching & cache-affecting upgrades (ch. 11, 33)

`docs/behavioral_contract.md` chapter **11** (Caching) was checked line-by-line against `CacheManager` (`_cache.py`) and `CheckOrchestrator._generate_cache_key()`/`process_files()` (`ast_checks/__init__.py`), following on from `0005`'s cache-key design and `0011`–`0013`'s prior audits. Chapter **33** (Compatibility and Upgrade Behavior) was checked only for its cache-invalidation-on-version-change portions, per issue #32's own scope — the remaining ch. 33 items (patch-release fix-semantics policy, config migration paths, large-diff documentation) are about CLI/config UX and release process, not caching, and are listed below as out of scope rather than analyzed in depth.

Most items were already satisfied, several by `0005` (source-tree-hash + per-check config fingerprint replacing the hand-bumped `CACHE_VERSION`) and `0013` (the `fcntl.flock` timeout, so a stuck peer degrades to a cache miss instead of hanging): cache hits are content-hash-verified before being trusted (mtime is only a fast-path, never trusted alone), every write is atomic (temp file + rename, with cleanup on error), a version mismatch discards the whole per-file blob rather than patching it in place (the `0e3efba` bug `0005` already fixed), corrupted JSON and a missing cache entry both degrade to a logged cache miss rather than a crash, and concurrent writers are serialized by an advisory lock. `--fix` mode already skips both cache reads and cache writes, so the closest thing this project has to a cache-disable switch already disables both sides of it consistently.

## Gaps found and fixed

- **`CacheManager.__init__` → `_ensure_cache_dir()` let `mkdir()`'s (or the `CACHEDIR.TAG` write's) `OSError` propagate uncaught** (ch. 11: "MUST tolerate cache directories being unavailable", "MUST gracefully degrade to uncached execution when caching is unavailable", "MUST NOT crash because the cache is missing, corrupted, partially written, or inaccessible", "MUST NOT assume that a cache directory is writable merely because it exists"). `CheckOrchestrator.__init__` constructs a `CacheManager` unconditionally with no surrounding `try`/`except`, so a cache directory that can't be created — permission denied, a read-only filesystem, a parent that doesn't exist and can't be created — crashed the entire hook with a raw traceback before a single file was checked, instead of running uncached. Reproduced concretely: constructing a `CacheManager` under a `chmod 0o555` parent directory raised `PermissionError` straight out of `__init__`. Fixed by wrapping `_ensure_cache_dir()`'s body in `try`/`except OSError`, logging the same kind of warning `get_cached_result()`/`set_cached_result()` already emit for their own `OSError`s. Two review passes refined this fix further:
  1. The first version stopped there and relied on those two methods' own existing `except OSError` to degrade each later call individually — caught in review as reintroducing ch. 11's "MUST warn when cache failures materially affect expected behavior or performance" as noise rather than a signal, and ch. 24/30's performance principles this project already treats seriously (`0013`): every processed file would repeat the same doomed `mkdir()` (and, in `set_cached_result()`, a full-file content hash computed _before_ even reaching that `mkdir()`) and log another warning, for the entire run. Fixed by adding `self._cache_dir_unavailable`, set once in `_ensure_cache_dir()`'s `except` branch, and checked first thing in both `get_cached_result()` and `set_cached_result()` to return a no-op immediately — one warning at construction, no wasted `stat()`/hash/`mkdir()` work per file thereafter.
  2. That version still missed a real case, caught in a second review pass: a cache directory that already exists from a prior run — `mkdir(parents=True, exist_ok=True)` succeeds trivially — with `CACHEDIR.TAG` already written — the `if not tag_file.exists(): tag_file.write_text(...)` guard skips the write entirely — never attempts a single write operation, so a directory later chmodded read-only (or a read-only mount reused across runs) raised nothing and left `_cache_dir_unavailable` `False`, silently falling back to the same per-file repeated-failure behavior (1) had just fixed. Fixed by adding an explicit `os.access(self.cache_dir, os.W_OK)` check after the mkdir/tag-write block, raising `PermissionError` (caught by the same `except OSError`) when it fails — a deliberate writability probe instead of inferring availability from whether a write happened to be attempted.

  Regression tests, all in `tests/test_cache.py`: `test_construction_does_not_crash_when_cache_dir_is_unavailable` (a `CacheManager` built under a read-only parent directory behaves as a no-op cache miss instead of raising), `test_unavailable_cache_dir_short_circuits_without_touching_filesystem` (with the flag set, `_get_cache_path()` is never called by either method), `test_construction_detects_pre_existing_read_only_cache_dir` (a cache dir built once successfully, then chmodded read-only and reopened by a second `CacheManager`, is correctly flagged unavailable).

- **`CheckOrchestrator._generate_cache_key()` never accounted for the running Python interpreter's version** (ch. 11: "MUST include the relevant target Python version in the cache identity"; ch. 33: "MUST invalidate caches when a version change can affect results"). Every check's results come from `ast.parse()`'s output, and that output isn't guaranteed identical for the same source text across Python minor versions (grammar and AST-shape changes between releases) — a `.cache/pre_commit_hooks/` directory shared across an interpreter upgrade (e.g. a contributor's local Python moving from one minor version to the next, which `AGENTS.md`'s own "support only the current stable Python version" policy anticipates happening over the project's lifetime) could silently reuse results computed under the old interpreter. Fixed by adding `f"{sys.version_info.major}.{sys.version_info.minor}"` as a fourth component of the joined cache key, alongside the existing check-id list, per-check config fingerprint, and source-tree hash. Deliberately major.minor only, not the full version string: bugfix releases don't change the grammar, so including patch/build metadata would invalidate the cache on every patch upgrade for no behavioral reason — the same "invalidate on real changes, not on noise" principle `0005` already applied to the tree hash. Regression test: `test_generate_cache_key_changes_when_python_version_changes` in `tests/test_orchestrator.py`, patching the `sys` name bound inside the `ast_checks` module (not the real global `sys` module, to avoid mutating process-wide state) to a fake minor version and asserting the generated key changes.

## Items judged not applicable, with rationale

- **Ch. 11, "MUST include all relevant configuration files in the cache identity"**: this project has no config-file parsing anywhere in the pipeline — grepped and confirmed empty (`pyproject.toml`, `tomllib`, `config_file`, `.toml` all absent from `src/pre_commit_hooks`). Every check's configuration comes exclusively from CLI arguments (which `.pre-commit-config.yaml`'s own `args:` list supplies), already captured in the cache key via `_fingerprint_check()`. Nothing to add here.
- **Ch. 11, "MUST include relevant environment-dependent inputs when the analysis depends on them"**: grepped for `os.environ`/`os.getenv`/`environ[` across `src/pre_commit_hooks` — no check reads an environment variable. Covered by construction, not by an explicit cache-key entry.
- **Ch. 11, "MUST ensure that cache cleanup cannot delete unrelated user files"**: there is no cache cleanup/pruning logic anywhere in `_cache.py` — a stale per-file blob for a since-deleted or renamed file is never removed automatically (same as `0013`'s ch. 24 finding on unbounded cache growth, matching `mypy`'s/`ruff`'s own convention of "delete the whole cache directory to clear it," which `CACHEDIR.TAG`'s own text already documents as the supported path). With no deletion code, there is nothing that could delete an unrelated file.
- **Ch. 11, "MUST ensure that disabling the cache truly disables cache reads and writes"**: there is no `--no-cache`/cache-disable CLI flag at all — grepped and confirmed absent. The closest analog, `--fix` mode, already gates both `_get_cached_violations()` and `_cache_violations()` behind the same `not self.fix_mode` condition (`ast_checks/__init__.py`), so reads and writes are already consistent with each other; there's no separate disable switch to audit for a read/write mismatch.
- **Ch. 33, "MUST define how behavior changes between tool versions"**: out of scope for a caching audit per issue #32's framing, and separately addressed by this project's own explicit policy (`AGENTS.md`: "Breaking changes are allowed and expected... don't design backward-compatibility shims, deprecation warnings, or migration paths") — a single-maintainer local hook with no external consumers and no semver stability promise, where git history and commit messages are the record of what changed, not a versioned compatibility policy document.
- **Ch. 33, "MUST avoid silently changing auto-fix semantics in a patch release unless the compatibility policy permits it"**: same rationale — this project has no patch/minor/major release process for the phrase "compatibility policy" to refer to.
- **Ch. 33, "MUST document behavior changes that can produce large source diffs"**: out of scope for a caching audit; satisfied structurally by this project's own practice of writing an ADR for every behaviorally significant change (this document being one), not something specific to caching.
- **Ch. 33, "MUST provide a migration path for incompatible configuration changes" / "MUST avoid silently interpreting old configuration according to a materially different meaning"**: out of scope for a caching audit (CLI/config UX, not cache invalidation), and explicitly disclaimed for this project's own CLI surface by `AGENTS.md`'s "don't design... migration paths for this project's own hook ids/CLI surface." Separately, there's no persisted config file for a CLI flag's meaning to silently drift underneath (see the config-files item above), so there's no "old configuration" for a later run to reinterpret.
- **Ch. 33, "MUST ensure that serialized internal data is versioned or safely invalidated when its format changes" / "MUST ensure that old cache data cannot silently produce incorrect results after an incompatible upgrade"**: already satisfied by `0005`'s design — `_cache.py`'s own serialization logic is inside `_PACKAGE_ROOT`, so any change to the cache blob's schema changes the source-tree hash, which changes `cache_version`, which makes `get_cached_result()` reject every existing on-disk blob (`cache_data.get("version") != self.cache_version`) before ever reading a field from it. Verified by the pre-existing `test_cache_version_mismatch`/`test_cache_version_mismatch_recovers_on_rewrite` tests.

## Consequences

- `CacheManager._ensure_cache_dir()` can no longer raise; an unavailable cache directory now logs one warning at construction instead of crashing, and gains a new `_cache_dir_unavailable` instance attribute that every later `get_cached_result()`/`set_cached_result()` call checks first to skip cache I/O entirely for the rest of that `CacheManager`'s life. A directory that becomes unavailable only _after_ successful construction (e.g. deleted mid-run) still falls through the pre-existing per-call `except OSError` path instead — unchanged, and already covered by `test_cache_write_errors_do_not_crash`.
- `CheckOrchestrator._generate_cache_key()`'s output now has four `|`-joined components instead of three; this invalidates every existing on-disk cache entry once (the key format itself changed), the same one-time cost every prior cache-key change in `0005`/`0011`–`0013` already carried.
- Running the same repository's hook under two different Python minor versions no longer risks sharing cached results between them, at the cost of a full cache miss the first time either interpreter's own entries expire after this change.
