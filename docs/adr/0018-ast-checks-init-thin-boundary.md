# `ast_checks/__init__.py` is a thin package boundary, not the linter's implementation

`ast_checks/__init__.py` had grown to 1060 lines, mixing `CheckOrchestrator`, the CLI (`main()`), file discovery (`expand_directories`/`filter_excluded_files`), and cache fingerprinting — everything the package does, in the one file Python treats as the package itself. Split into `_cli.py` (argparse plus the exit-code contract), `_discovery.py` (CLI filename args → concrete file list), `_diagnostics.py` (the violation/failure report), and `_orchestrator.py` (`CheckOrchestrator`, `load_checks`, cache fingerprinting). `__init__.py` now holds only the module docstring and the `ALL_CHECKS` registry.

The new modules are underscore-prefixed (`_cli.py`, not `cli.py`) to match this package's existing convention for internal infrastructure (`_base.py`, `_scope.py`, `_cache.py`, `_prefilter.py`) rather than a generic layout template's plain names — the only non-underscored modules here are check/rule implementations (`forbid_vars.py`, `misplaced_comment.py`, ...), which are this package's actual public unit. `__init__.py` does not re-export the moved symbols; `CheckOrchestrator`, `load_checks`, `main`, `expand_directories`, and `filter_excluded_files` are now imported from their concrete modules everywhere, including in tests, rather than kept reachable from the package root for compatibility.
