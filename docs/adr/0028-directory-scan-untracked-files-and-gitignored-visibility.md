# Directory-argument discovery includes untracked files; gitignored exclusions are now visible

`_prefilter.py`'s `git_grep_filter()` always searches an explicitly-named file regardless of its tracked/ignored status (ADR 0024), but `_discovery.py`'s `_list_python_files_in_dir()` expanded a directory argument via plain `git ls-files`, which lists only tracked files. ADR 0024 judged this asymmetry "not the same gap"; ADR 0027's synthesis pass revised that conclusion — naming an untracked file explicitly caught its violation, while naming the containing directory instead silently skipped the identical file with no diagnostic, the same false-clean result ADR 0024 fixed for the explicit-file path. Filed as [issue #67](https://github.com/alessio-locatelli/ruff-extra-rules/issues/67).

## Decision

`_list_python_files_in_dir()` now runs `git ls-files --cached --others --exclude-standard` instead of plain `git ls-files`, adding untracked-but-not-`.gitignore`d files to a directory scan's results. This resolves the asymmetry for the common case the issue reproduces: a brand-new file that hasn't been `git add`ed yet is now in scope whether it's named directly or reached by expanding its containing directory.

A genuinely `.gitignore`d file is still excluded from a directory scan — ADR 0015's rationale (avoid sweeping `.venv`/build artifacts that happen to live under the scanned directory into every check) still holds, and matching `git_grep_filter`'s `--no-exclude-standard` behavior here would reintroduce exactly that sweep-in. Instead, `_warn_about_ignored_python_files()` runs `git status --porcelain=v1 --ignored --untracked-files=normal` scoped to the directory and logs one `logger.warning()` if it reports any ignored `.py` file or ignored directory, so the exclusion is now visible rather than silent. `--untracked-files=normal` pins git's own default explicitly rather than trusting it: a user's `status.showUntrackedFiles=no` config would otherwise suppress every ignored-file record and silently defeat this warning's whole purpose, while `status.showUntrackedFiles=all` would force recursion into an entirely-ignored directory instead of collapsing it to one line. With the default pinned, an entirely-ignored directory collapses to a single line instead of recursing into it — unlike `git ls-files --others --ignored`, which would walk every file inside e.g. `.venv/` and reintroduce the same cost this decision otherwise avoids — so the warning can't always confirm an ignored directory actually contains a `.py` file, and reports it anyway rather than staying silent on that uncertainty.

## Consequences

- `_list_python_files_in_dir()` issues one extra `git status` call per directory argument to detect ignored content worth warning about. An entirely-ignored directory (e.g. `.venv/`) collapses to one line regardless of its size, but a directory that's only _partially_ tracked (a generated/ subtree with one tracked file alongside many individually-`.gitignore`d outputs) can't collapse the same way — git still reports each ignored path separately, and `subprocess.run(capture_output=True)` has to buffer that entire reply before this function's own `_MAX_REPORTED_IGNORED_PATHS` (20) cap can bound what's actually logged. A fully bounded fix would need to stream and early-terminate the `git status` process instead of buffering its whole output — judged disproportionate engineering weight for a purely supplementary diagnostic in a solo hobby project (`AGENTS.md`) with no such directory shape in its own tree. Mitigated, not eliminated: this probe gets a 5-second timeout (vs. 30s for the load-bearing discovery calls), so a pathological tree degrades to "no warning printed" — the directory scan's own result is already correct without this warning — rather than materially slowing down the run.
- A directory scan's result set now differs from before whenever an untracked, non-ignored `.py` file exists under it — this is the intended fix, not a side effect.
- `docs/adr/0024` and `docs/adr/0027` both point forward to this decision as the resolution of issue #67.
- Both git subprocess calls in `_discovery.py`, plus `_prefilter.py`'s pre-existing `git_grep_filter()` call, now pass `errors="surrogateescape"` alongside `text=True`. A filename that's a valid path on Linux but not decodable under the locale's preferred encoding previously raised `UnicodeDecodeError` straight out of `subprocess.run()` and crashed the run; `surrogateescape` — the same handler `os.fsdecode()` already uses for filesystem paths — never raises and round-trips back to the exact file. Falling back to the untracked, non-`.gitignore`-aware rglob scan instead (mirroring every other subprocess failure this module already handles) was considered and rejected: it would silently reintroduce the `.venv`/build-artifact sweep-in this ADR's own directory-scan fix avoids, for the entire directory, merely because one filename in it was oddly encoded.
