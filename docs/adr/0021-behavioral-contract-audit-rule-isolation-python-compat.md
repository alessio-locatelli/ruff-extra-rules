# Behavioral contract audit: rule isolation & Python-version compatibility (ch. 18, 19)

`docs/behavioral_contract.md` chapters **18** (Rule and Plugin Isolation) and **19** (Python Version and Language Compatibility) were checked line-by-line against `ASTCheck`/`atomic_write_text()` (`ast_checks/_base.py`), `CheckOrchestrator`/`load_checks()`/`_generate_cache_key()` (`ast_checks/_orchestrator.py`), `ALL_CHECKS` (`ast_checks/__init__.py`), `_cli.py`'s CLI wiring, and every check's own `fix()` (`forbid_vars.py`, `excessive_blank_lines.py`, `misplaced_comment.py`, `redundant_assignment/autofix.py`, `validate_function_name/autofix.py`), following on from `0011`–`0020`'s prior audits, per issue #36.

Both chapters are written for a linter with a dynamic or third-party plugin system and a configurable analysis target version — this project has neither. There is no plugin mechanism at all: `ALL_CHECKS` (`ast_checks/__init__.py:44`) is a fixed, hand-authored `list[type[ASTCheck]]` of in-repo classes, with a comment stating exactly that ("no plugin mechanism for third-party checks, so a static list is all that's needed"). And there is no separate "target Python version" concept independent of the interpreter actually running the hook: `AGENTS.md` states this project "support[s] only the current stable Python version (currently `>=3.14`)", `pyproject.toml`'s `requires-python = ">=3.14"` and `[tool.ruff] target-version = "py314"` say the same for the project's own source, and grepping confirms no `ast.parse`/`compile` call anywhere in `src/` passes a `feature_version` or otherwise selects a grammar different from the running interpreter's own. Every syntax-related item in chapter 19 collapses to "whatever the running interpreter's own `ast`/`compile` accepts," which most of this audit's chapter-19 findings below trace back to.

## Already satisfied, confirmed with citations

- **Ch. 18, "MUST define the interface between rules and the core engine"**: the `ASTCheck` `Protocol` (`_base.py:41-119`) is that interface — `check_id`, `error_code`, `get_prefilter_pattern()`, `check()`, `fix()`, `add_cli_arguments()`/`cli_kwargs_from_args()` — and `docs/adding-a-check.md` documents it a second time for a human author, including the constraint (`adding-a-check.md:129`) that "`check()` must only ever look at the single file it's given (`filepath`/`tree`/`source`) — no check reads another file, imports, or any other cross-file state."
- **Ch. 18, "MUST ensure that a rule cannot accidentally modify source outside its declared scope"**: every check's only write path is `atomic_write_text(filepath, ...)`, targeting exactly the one `filepath` the check itself was given — grepped every `fix()` implementation (`forbid_vars.py:421`, `excessive_blank_lines.py:261`, `misplaced_comment.py:227`, `redundant_assignment/autofix.py:135`, `validate_function_name/autofix.py:366`) and confirmed none performs any other file, network, or environment I/O (`redundant_super_init.py` has no autofix at all). Within a file, `forbid_vars.py`'s multi-violation renamer (`_apply_fixes()`, `forbid_vars.py:359-421`) additionally resolves each replacement's scope fresh against the current tree and applies edits in descending `(line, col)` order specifically so an earlier edit never shifts a not-yet-applied position — the general pattern the other checks' own fix logic (already audited in depth by `0011` for chapters 1/3/4/20) follows too.
- **Ch. 18, "MUST validate rule-provided edits before applying them"; "MUST apply global validation to all fixes regardless of their source"; "MUST NOT allow a plugin to silently bypass global safety checks"**: `atomic_write_text()` (`_base.py:242-292`) is the one function every check's `fix()` writes through, and it always `compile()`s the proposed content before any file I/O — a fix from any check that would produce invalid syntax is rejected (`FixValidationError`) before a temp file even exists. No check has, or could have, a second write path that skips this (previous bullet).
- **Ch. 18, "MUST handle malformed rule output safely"; "MUST isolate failures in third-party or dynamically loaded rules where practical"; "MUST NOT allow a broken plugin to corrupt the core linter state"**: `_check_file()`'s per-check loop (`_orchestrator.py:288-297`) and `_apply_fixes()`'s per-check loop (`_orchestrator.py:330-464`) each wrap a check's `check()`/`fix()` call in its own `try`/`except Exception`, recording `(filepath, check_id)` in `rule_failures` and letting every other check and file continue unaffected — audited in depth already by `0012` for chapters 5/6. "Third-party or dynamically loaded" doesn't apply (intro paragraph), but the isolation itself is unconditional — it isolates every check's failure, in-house or not.
- **Ch. 18, "MUST identify the rule responsible for a failed diagnostic or fix"**: `rule_failures: list[tuple[str, str]]` (`_orchestrator.py:107`) records the failing `check_id` alongside the file, and `_diagnostics.py:35-41` prints it by name — also `0012`.
- **Ch. 18, "MUST define compatibility requirements for plugins"**: no third-party plugin ABI exists to version (intro paragraph), but `docs/adding-a-check.md`'s "Requirements" section (lines 50-58) is the equivalent contract for an in-house check: standard-library-only, never touch text inside string/comment literals, support `# pytriage: ignore=`, write only through `atomic_write_text()`, and the specific fix-outcome contract (`True`/`False`/`FixValidationError` propagation) `0010`/`0011`/`0012` each hardened.
- **Ch. 18, "MUST invalidate caches when rule implementation behavior changes"**: `_generate_cache_key()` (`_orchestrator.py:174-198`) hashes every enabled check's `check_id`, a fingerprint of its own constructor arguments, a SHA-1 of this whole package's own source tree, and the running interpreter's `major.minor` — any change to a check's code invalidates every cached result for every check, deliberately coarse-grained (`0005`, `0014`).
- **Ch. 18, "MUST ensure that disabled rules do not produce diagnostics or fixes"**: `load_checks()` (`_orchestrator.py:580-619`) only instantiates and returns checks that pass the `select`/`ignore` filter — a disabled check is never in `self.checks` at all, so it's never called.
- **Ch. 18, "MUST ensure that rule selection is deterministic"**: `load_checks()` iterates `ALL_CHECKS` in its fixed, hand-authored order and filters by set membership; the resulting `checks` list's order never depends on `--select`/`--ignore`'s own argument order (both parsed into unordered `set[str]`, `_cli.py:103-104`).
- **Ch. 19, "MUST define the supported Python language versions"**: `pyproject.toml`'s `requires-python = ">=3.14"` and `classifiers` (`Programming Language :: Python :: 3.14`), plus `AGENTS.md`'s explicit "Support only the current stable Python version (currently `>=3.14`)."
- **Ch. 19, "MUST analyze syntax according to the configured target version"; "MUST NOT silently interpret newer syntax as older syntax incorrectly"**: there's exactly one "configured target version" — the interpreter actually running the hook — so `ast.parse()` (`_orchestrator.py:279`) always parses using that interpreter's own grammar; there is no separate, possibly-mismatched target to silently misinterpret against, and no `feature_version`/downlevel-grammar code path exists anywhere in `src/` (grepped, confirmed empty) that could apply the wrong one.
- **Ch. 19, "MUST reject or clearly report unsupported syntax"**: a `SyntaxError` from `ast.parse()` is caught by `_check_file()` (`_orchestrator.py:278-285`) and the file is added to `unprocessable_files`, reported as `"{filepath}: error: could not be read or parsed; file skipped"` with `exit_code = 1` (`_diagnostics.py:28-30`) — audited in depth already by `0011` for chapter 4's identical "invalid Python" requirement; chapter 19 asks for the same thing from the angle of "syntax outside the supported version," which is a subset of "invalid Python for this interpreter" given the one-target-version design above.
- **Ch. 19, "MUST ensure that auto-fixes are valid for the configured target version"; "MUST NOT apply a fix that introduces syntax unsupported by the target Python version"**: `atomic_write_text()`'s `compile()` call (`_base.py:278`) already validates every fix against the running interpreter before writing — and since the running interpreter is the one and only "target version" (intro paragraph), that validation already is the target-version check this item asks for.
- **Ch. 19, "MUST ensure that version-dependent rules use the configured target version consistently"**: grepped every check for `sys.version_info` or an equivalent version branch (`rg -n "sys.version_info|version_info" src/`) — the only hit is `_generate_cache_key()`'s own cache-identity computation (next bullet); no check's detection or fix logic branches on Python version at all, so there is nothing version-dependent to keep consistent.
- **Ch. 19, "MUST include the target version in relevant cache identity"**: `_generate_cache_key()` includes `f"{sys.version_info.major}.{sys.version_info.minor}"` (`_orchestrator.py:197`), with an existing comment explaining why (AST shape isn't guaranteed stable across interpreter minor versions, so a `.cache` directory shared across an upgrade mustn't reuse stale results) and why only `major.minor` (bugfix releases don't change the grammar).
- **Ch. 19, "MUST distinguish the Python version used to run the linter from the Python version being analyzed"**: this project defines the two to be identical rather than conflating two distinct axes — there is no CLI flag, config key, or per-file directive anywhere that selects an analysis target different from the running interpreter (grepped `_cli.py` and every check's `add_cli_arguments()`; only `--select`/`--ignore`/`--fix`/`--exclude`/`--list-checks` plus each check's own narrow options exist). "Distinguish" doesn't require two different values to exist, only that the tool not silently confuse them if they could differ — moot here since they can't.
- **This project's own use of PEP 758 (`except A, B:`, no parentheses) syntax is itself a live, continuously-exercised proof of the one-target-version design working correctly**: `src/pre_commit_hooks/_prefilter.py:123`, `ast_checks/_orchestrator.py:268`, `ast_checks/_discovery.py:109`, `ast_checks/__main__.py:44`, and `ast_checks/redundant_assignment/analysis.py:378` all use it. `.pre-commit-config.yaml`'s own `ruff-extra-rules` hook entry runs this project's checks against `src/` and `tests/` on every commit (`files: ^(src|tests)/`, excluding only `tests/fixtures/`) — meaning every one of those PEP 758 clauses is itself parsed and walked by `forbid_vars`/`redundant_assignment`/etc. on every commit without incident, since PEP 758 produces the identical `ast.Tuple` shape for `ExceptHandler.type` as the parenthesized form and no check does any text/regex-based parsing of `except` clauses (grepped, confirmed AST-only). Likewise `_orchestrator.py:41`'s own `type ViolationKey = tuple[int, int, str]` (PEP 695) is dogfooded the same way.

## Not applicable, with rationale

- **Ch. 18, "MUST isolate failures in third-party or dynamically loaded rules where practical"; "MUST define compatibility requirements for plugins"**: partially answered above (isolation and in-house compatibility requirements do exist), but the literal "third-party"/"dynamically loaded" framing doesn't apply — `ALL_CHECKS` is a compile-time-fixed list of classes defined in this same package, never discovered via an entry-point, plugin directory, or dynamic import.
- **Ch. 19, "MUST test rules across all supported language versions where behavior differs"**: there is exactly one supported language version (`AGENTS.md`), so there is no cross-version behavior difference to test — the same "single supported platform, no matrix to run" rationale `0013`/`0016`/`0020` already gave for their own chapters' CI-matrix-shaped items.

## Regression test added

No behavior in either chapter was actually wrong — the audit didn't reproduce a single case of two checks sharing a `check_id`/`error_code`, a fix bypassing `atomic_write_text()`, or a version mismatch, so nothing above is a "gap found and fixed" in the sense `0011`-`0020` used that phrase. One item was true today but unverified: `docs/adding-a-check.md` already documents "next unused number" as the convention for picking a new check's `check_id`/`error_code` (ch. 18: "MUST define how duplicate rule identifiers are handled"; "MUST NOT silently resolve conflicting rule identifiers in an ambiguous way"), but nothing enforced it — a future check added with a copy-pasted or mistyped `check_id` would silently run alongside the check it collided with, indistinguishable to `--select`/`--ignore` (both do plain `set` membership against `check_id`) and to `_generate_cache_key()`'s own sorted `check_id` list. `test_all_checks_have_unique_check_ids_and_error_codes` (`tests/test_orchestrator.py`) locks the current, already-correct state in going forward: it fails loudly the moment a duplicate is introduced, rather than requiring a live run to notice two checks behaving ambiguously under the same identifier. This is a preventive test, not a fix — there is no prior commit against which it would have failed.

## Consequences

- `tests/test_orchestrator.py` gains `test_all_checks_have_unique_check_ids_and_error_codes`, asserting `ALL_CHECKS` never collides on `check_id` or `error_code`. No production code changed.
- No behavior change for any existing check, CLI flag, cache key, or fix path.
