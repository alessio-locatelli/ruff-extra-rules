# Behavioral contract audit: diagnostics, auto-fix modes & user trust (ch. 7, 8, 34)

`docs/behavioral_contract.md` chapters **7** (Diagnostics and User Feedback), **8** (Auto-Fix Modes), and **34** (User Trust) were checked line-by-line against `CheckOrchestrator` and `main()`'s own reporting (`ast_checks/__init__.py`), the four checks with their own file-write logic (`excessive_blank_lines.py`, `misplaced_comment.py`, `forbid_vars.py`, `redundant_assignment/autofix.py`), and every `logger.exception()` call site across `src/`, following on from `0011`–`0016`'s audits of chapters 1, 3–6, 11–13, 15, 16, 20, 24, 25, 30, and 33, per issue #41.

Most items were already satisfied: check mode never writes (`_apply_fixes` is only called when `fix_mode` is `True`), `[FIXABLE]`/`[FIXED]`/`[FIX REJECTED]`/`[FIX ERRORED]` already gave a partially-fixable file's diagnostics a clear, per-violation outcome, `main()` never claims every violation was fixed, and every diagnostic already carries a file, line, rule identifier, and human-readable message.

## Gaps found and fixed

- **Every diagnostic's own column was silently dropped, and — once added — two of the checks reporting one were reporting the wrong number.** `main()`'s printed line only ever interpolated `v.line`, never `v.col` (ch. 7: "MUST report line and column information accurately when available"). Fixed by adding the column to the printed line — but a review caught that simply printing `v.col` as-is was itself inaccurate for two of the checks that populate it: `forbid_vars` and `redundant_assignment` both stored `ast.col_offset` directly, which is a UTF-8 _byte_ offset, not a character offset (`_base.py`'s own pre-existing `byte_col_to_char_col()`, already used by both checks' own fix logic for exactly this reason, documents the distinction) — a line with non-ASCII text before the violation reported a column too far right. `misplaced_comment`'s own `tokenize`-derived column was already a character offset, needing no change. Fixed by converting `forbid_vars`' and `redundant_assignment`'s own diagnostic-facing column with `byte_col_to_char_col()` at the one place each actually constructs its `Violation` (their own fix logic already re-derives its edit positions fresh from the tree rather than reading this stored value back, so this only affects what's reported, never what's edited). `main()` also now reports the conventional 1-based column (`v.col + 1`) instead of the AST's own native 0-based one, matching `ruff check`'s own convention (this project's own explicit CLI parity target, `0008`) and removing the ambiguity between "column 0, a real position" and "column 0, this check tracks no more specific position than the line" (`excessive_blank_lines`/`redundant_super_init`/`validate_function_name` all use a plain `0` placeholder for the latter). Regression tests: `test_main_reports_column_alongside_line` (updated), `test_check_reports_character_offset_not_byte_offset_before_multibyte_text` in `tests/test_forbid_vars.py` and `tests/redundant_assignment/test_check.py`.

- **`ASTCheck.fix()`'s own documented `True`/`False`-never-raises contract for a caught `OSError` was never actually inspected by its only caller.** `_base.py`'s own docstring already told every check to catch an `atomic_write_text()` `OSError` (disk full, permission denied, missing parent directory) itself and return `False` rather than let it propagate — and `excessive_blank_lines`, `misplaced_comment`, `forbid_vars`, and `redundant_assignment` all already did exactly that. But `CheckOrchestrator._apply_fixes` never read the return value at all: after a caught `OSError`, the violation was left in its original, unmarked state and reported as ordinary `[FIXABLE]` with "Run with --fix to inline automatically" — actively misleading, since `--fix` had already been attempted and had already failed for a reason a bare retry won't fix (ch. 7: "MUST clearly report when a fix was not applied"; ch. 8: "MUST define how a partially fixable file is handled"; ch. 34: "MUST NOT silently ignore failures"). This is the same class of gap `0012` already fixed for an _unexpected exception_ during `fix()` — but that fix never covered this separate, already-documented "caught its own I/O error" outcome. Fixed by completing the three-way contract `_base.py`'s docstring already implied: a new `mark_fix_failed()`/`is_fix_failed()` pair (`_base.py`, mirroring `mark_fix_rejected()`/`mark_fix_errored()`), wired into each of the four checks' own `except OSError` branch (marking every violation the failed write would have covered — the whole batch for a single-write check, or `redundant_assignment`'s own `applied_violations` subset specifically), and a new `[FIX FAILED]` tag in `main()`'s report loop with its own hint ("check file permissions and available disk space, then run with --fix again") — deliberately _not_ reusing `[FIX ERRORED]`'s "this is a bug, please report it" wording, since a permission-denied or disk-full condition isn't a bug in the check's own logic the way an unexpected exception is (ch. 7: "MUST provide a useful human-readable message"). Regression tests: `test_mark_fix_failed`/`test_is_fix_failed_false_when_only_marked_errored` in `tests/test_base.py`; `test_fix_write_failure_returns_false` (extended) in each of `tests/test_excessive_blank_lines.py`, `tests/test_misplaced_comment.py`, `tests/test_forbid_vars.py`, `tests/redundant_assignment/test_autofix.py`; `test_main_fix_flag_reports_failed_fix` in `tests/test_orchestrator.py`.

  Two further, structurally identical gaps turned up in `_apply_fixes` itself while fixing the above: its mid-loop re-read (`self._read_source(filepath)`, done again before each fixable check's own recompute, in case an earlier check in the same loop already modified the file) and its own broad outer `except Exception` (covering the re-parse and the fresh-violations recompute) both `continue`d or logged with _zero_ user-facing signal — no `rule_failures` entry, no violation marking, nothing beyond an ad hoc `logger.exception()` call. A check whose re-read or recompute failed this way kept its stale pre-fix snapshot and was reported as ordinary `[FIXABLE]`, exactly as misleading as the `OSError` gap above. Fixed the same way `0012` already fixed the equivalent gap for `fix()` itself raising: both paths now append to `self.rule_failures` and call `mark_fix_errored()` on the check's own still-fixable stale entries. Regression tests: `test_main_reports_rule_failure_when_reread_fails_mid_fix_loop`, `test_main_reports_rule_failure_when_recompute_raises_mid_fix_loop` in `tests/test_orchestrator.py`.

- **A violation left open after its own check's `fix()` ran never got its own stale, pre-fix line number refreshed again, even when a later check's fix in the same run shifted every line after it.** `_apply_fixes`'s existing recompute-before-fix already handled the _fixable-vs-fixable_ case (documented by `0011`), but only once, immediately before that check's own `fix()` call — never again afterward. Two distinct shapes of this: a check that's never fixable at all (`redundant-super-init`; `RedundantSuperInitCheck.fix()` always returns `False`) never enters the per-check loop in the first place, so its very first, pre-any-fix `check()` result is carried through to the final report completely untouched; and a check that _did_ run its own `fix()` this call but left one of its own violations open (e.g. `validate-function-name`'s `should_autofix` guard refusing to touch a method while renaming an unrelated free function in the same call) has that specific violation frozen at whatever position it had when _that check's own_ recompute ran, not the file's true final state. Reproduced both ways: `redundant-super-init` alongside `redundant-assignment` (the former never participates at all), and `validate-function-name` alongside `redundant-assignment` where the method violation is left open by the same `fix()` call that successfully renamed a sibling function (ch. 7: "MUST report line and column information accurately when available"; ch. 34: "MUST NOT silently use stale analysis results when they can affect correctness"). Fixed with a final reconciliation pass, `CheckOrchestrator._refresh_stale_positions()`, run once after the per-check loop finishes — but only when the loop actually resolved at least one violation this call (tracked via a new `file_changed` flag, itself derived from `_mark_resolved_and_get_still_present()`'s own return value), so a run where every fix attempt was rejected or failed outright doesn't pay for an extra read+parse+recheck for nothing. The pass considers each check_id's whole current group of entries together: if _any_ of them already carries a `fix_rejected`/`fix_errored`/`fix_failed` mark, that check_id is skipped entirely this pass, its still-open entries left exactly as they were — a fresh `check()` call would rediscover that same still-present rejected/errored/failed violation too, and there's no reliable way to tell that rediscovery apart from a genuinely different, unrelated violation that merely happens to share message text with it (e.g. two identically-named functions in different scopes), for lack of a stable per-violation identity that survives a `check()` call in this codebase; an earlier version of this fix tried to filter the fresh result by message instead, which a review caught silently dropping exactly that kind of unrelated violation from the report entirely — worse than leaving a position stale (ch. 34: "MUST prefer a visible failure over a silent incorrect result"). Only when a check_id has no rejected/errored/failed entries at all does the pass drop its still-open (non-`fixed`) entries and replace them with that check's fresh `check()` result against the file's true final state; an entry already marked fixed is always left untouched regardless, since it's genuinely gone from the file and a fresh `check()` call would never rediscover it anyway. Regression tests: `test_apply_fixes_refreshes_non_fixable_checks_position_too`, `test_apply_fixes_refreshes_a_participating_checks_own_left_open_violation`, `test_refresh_stale_positions_never_drops_an_unrelated_open_violation_sharing_a_check_id_with_a_terminal_one` in `tests/test_orchestrator.py`.

- **Several internal `logger.exception()` calls leaked a full raw Python traceback onto the ordinary user's stderr by default, duplicating and cluttering the clean diagnostic line the hook already prints for the same failure.** Nothing in this codebase configures logging (no `basicConfig()`, no handler, no level anywhere), so every `logger.warning()`/`logger.exception()` call falls through to Python's own `logging.lastResort` handler, which prints any `WARNING`-or-above record straight to `sys.stderr` by default — confirmed by running the hook as a real subprocess (not through pytest, whose own logging-capture plugin masks this from every other test in this suite) against a permission-denied file: three separate tracebacks, one from `_prefilter.py`'s own git-grep-unavailable fallback and two duplicates from `_read_source`, all printed _before_ the tool's own single, documented `file: error: could not be read or parsed; file skipped` line (ch. 7: "MUST NOT emit uncontrolled human-oriented text into a machine-readable output stream"; ch. 7: "MUST clearly distinguish diagnostics ... from ... informational messages"; ch. 34: "MUST make errors actionable" — a wall of internal traceback isn't actionable to an ordinary user). Fixed by downgrading every `logger.exception()` call whose failure is _also_ independently and cleanly reported elsewhere (directly, or after the `[FIX FAILED]`/silent-path fixes above) to `logger.debug(..., exc_info=True)` — preserving the traceback for a developer who explicitly raises the log level, while no longer polluting the ordinary user's stderr by default. This is the same "developer-only" convention `_base.py`'s own pre-existing `find_ignored_lines()` already used for its own defensive `tokenize.TokenError` branch. Two of `_prefilter.py`'s/`ast_checks/__init__.py`'s calls are purely self-healing fallbacks (git unavailable → equivalent Python-only scan; unreadable file during pre-filtering → the file is kept as a candidate and the hook's own downstream read reports the real failure) with no separate "genuine failure" to distinguish at all. `ruff`'s own `BLE001` ("do not catch blind exception") rule specifically recognizes a `logger.exception(...)` call as evidence an `except Exception:` isn't swallowing its error, but doesn't recognize `logger.debug(..., exc_info=True)` the same way; each of the five affected `except Exception:` blocks got a one-line `# noqa: BLE001` explaining why (already caught, isolated per ch. 5, and logged — just at debug level). `load_checks()`'s own two `logger.exception()` calls (a check's `__init__` raising) were deliberately left unchanged: `0012` already examined this exact path and decided not to extend the exit code for it, reasoning it's effectively unreachable with this project's fixed, plugin-free check list; downgrading its only remaining signal without also revisiting that decision would make it strictly worse, not better. Regression tests: `caplog`-based level assertions added throughout the existing exception-path tests in `tests/test_prefilter.py`, `tests/test_orchestrator.py`, `tests/test_excessive_blank_lines.py`, `tests/test_misplaced_comment.py`, `tests/test_forbid_vars.py`, `tests/redundant_assignment/test_autofix.py`, and `tests/validate_function_name/test_check.py`; `test_real_invocation_does_not_leak_a_traceback_onto_stderr` in `tests/test_main.py` is the one true end-to-end proof, since it's the only test in the suite that spawns a real subprocess rather than calling `main()`/`run()` in-process (verified to fail against the pre-fix code before being merged).

## Items judged not applicable, with rationale

- **Ch. 7, "MUST provide a machine-readable output mode suitable for CI and editor integrations"; ch. 34, "MUST ensure that editor integrations can reliably determine the current state of diagnostics"**: this hook's one output format (`file:line:col: code: [tag] message`, one diagnostic per stderr line) is already the traditional compiler/grep-style location format most CI "problem matcher" and editor "quickfix" integrations already parse; it isn't a JSON/structured mode like `ruff check --output-format=json`. `0008` already deliberately deferred every `ruff check` flag beyond `--select`/`--ignore`/`--fix`/`--exclude`/`--list-checks` "to avoid scope creep," explicitly naming `--diff`/`--show-fixes`/`--statistics` as the kind of flag out of scope — a structured output mode is the same category of new CLI surface, not a fix to existing behavior. This project also claims no editor-integration surface today (only pre-commit/prek, which needs the exit code and stderr text this hook already provides reliably).
- **Ch. 7, "MUST provide an appropriate quiet mode where supported"**: not promised anywhere in this hook's own CLI surface (`--list-checks`/`--select`/`--ignore`/`--fix`/`--exclude`, per `0008`), so there's no quiet mode to keep honest.
- **Ch. 8, "MUST make unsafe or unavailable fixes distinguishable from safe fixes"**: this hook has no `--unsafe-fixes`-style concept at all — `0008`'s own consequences already document that every check "gates its own fix safety internally (complexity/collision checks)" before a fix is even attempted, so every fix this tool ever proposes is unconditionally the safe kind by construction, not one of two tiers a user has to distinguish.
- **Ch. 8, "MUST make the relationship between fix mode and formatting mode explicit when both exist"**: already explicit at the project level, not just undocumented-but-true — `CONTEXT.md`'s own opening line and `AGENTS.md`'s own title both state this hook "runs alongside ruff, not instead of it," and `ruff-format` is a wholly separate hook in `.pre-commit-config.yaml`. This hook owns no general reformatting of its own; `excessive_blank_lines`/`misplaced_comment`'s fixes are lint-rule-driven, not a competing formatter.
- **Ch. 8, "MUST NOT allow independent tools or passes to continually undo each other's changes without detecting the conflict"**: a single pre-commit hook has no visibility into what a _different_, externally-invoked hook (like `ruff-format`, run as its own separate process) does to the same file — this can't be detected from inside this hook's own process. pre-commit/prek's own convention of re-staging and re-running every hook until the working tree stops changing is the actual safety net here, the same one `ruff-format`/`ruff check --fix` themselves rely on; this hook's own fixes are already required to be idempotent (spot-checked by `0011`), which is what makes that convention converge rather than loop.
- **`validate_function_name`'s own internal `should_autofix()`/`apply_fix()` `except (OSError, SyntaxError, UnicodeDecodeError, LookupError)` branches were not wired into `mark_fix_failed()`.** Unlike the four checks fixed above, these two functions mix a genuine I/O failure with several purely structural, by-design declines behind the same plain `False` return (`should_autofix()`'s complexity/size/nesting gates; `apply_fix()`'s "function no longer found at the recorded position," "name rebound in scope") — conflating them would tag a legitimate, unrelated safety-guard decision with a "check permissions and disk space" hint that doesn't apply to it. These two functions already log via `logger.warning()`, not `logger.exception()`, so they were never part of the traceback-leak problem this audit's logging fix addresses — the message is already a clean, visible, single line. Disentangling the two would need a more invasive refactor of this module's own boolean-only API, disproportionate to what this audit's scope (chapters 7, 8, 34) requires; left as a narrower, deliberately deferred follow-up rather than folded into this pass.

## Consequences

- `Violation`'s reported diagnostics now include the column (`file:line:col: code: message`), 1-based and always a character offset, matching the location format most CI/editor tooling (and `ruff check` itself) already expects from a compiler-style diagnostic line. `Violation.col`'s own docstring now states this contract explicitly; `forbid_vars` and `redundant_assignment` convert their own `ast.col_offset` before storing it.
- `_base.py` gains a third fix-outcome marker, `mark_fix_failed()`/`is_fix_failed()`, alongside the existing `mark_fixed()`/`mark_fix_rejected()`/`mark_fix_errored()` trio; `Violation.fix_data` can now carry a `fix_failed` key. `main()`'s report loop gains a corresponding `[FIX FAILED]` tag and hint, checked after `[FIX ERRORED]` and before the ordinary `[FIXABLE]` case.
- `CheckOrchestrator._apply_fixes` gains a `file_changed` tracking flag and calls a new private method, `_refresh_stale_positions()`, once per fixed file when at least one violation was actually resolved this call — a small, targeted extra read+parse+recheck cost paid only on files `--fix` actually changed, not on every file or every `--fix` invocation. It deliberately trades some precision for safety: a check_id with any rejected/errored/failed entry is left entirely alone rather than risk silently dropping an unrelated violation that happens to share message text with it.
- Several `logger.exception()` calls throughout `src/` are now `logger.debug(..., exc_info=True)` — still fully available to a developer who raises the log level explicitly, but no longer visible in a plain, unconfigured invocation's stderr by default. `load_checks()`'s two `logger.exception()` calls for a check's own `__init__` raising are the one deliberate exception, per `0012`'s own prior reasoning.
