# Extend self-checking to tests/, reconciling violations rather than excluding them

`.pre-commit-config.yaml`'s `ruff-extra-rules` hook ran only against `src/` (`files: ^src/`), excluding `tests/` entirely from this repo's own self-dogfooding. The exclusion was never scoped, only deferred: `docs/work-items/10-lint-tests-directory.md` flagged it as an open maintainer decision, then put it on hold pending the CLI redesign tracked by `docs/adr/0008-ruff-check-cli-parity.md` (`docs/work-items/archived/11-ruff-check-cli-parity.md`), since scoping it against a CLI surface that was about to change would have been wasted effort. That redesign has since landed.

With the blocker gone, the open question was scoped empirically rather than by assumption: running all 6 checks against `tests/*.py` (excluding `tests/fixtures/`) showed violations concentrated in exactly 2 of 6 checks. `forbid-vars` (TRI001) fired 25 times, always on the literal name `result`. `redundant-assignment` (TRI005) fired 255 times, overwhelmingly on one repeated idiom — `tree = ast.parse(source)` followed by `check = XCheck()`, each fed into the next line and never reused. The other four checks (`excessive-blank-lines`, `misplaced-comment`, `redundant-super-init`, `validate-function-name`) were already 100% clean against `tests/`. `tests/fixtures/` (not just the `bad/` subdirs the module docstrings call out — `good/`, `ignore/`, and `autofix/` fixtures also incidentally tripped unrelated checks) stays excluded regardless, since it's deliberately non-representative example code, not a signal about test-suite quality.

## Considered Options

- **Keep `tests/` excluded entirely, close the work item as won't-fix**: rejected — the friction was concentrated in two checks and was mostly mechanical (line-length- and call-shape-limited, not semantic disagreement with the checks' judgment), not evidence the checks are wrong for test code. 198 of 280 violations resolved via plain `--fix`.
- **Extend only the 4 already-clean checks; leave `forbid-vars`/`redundant-assignment` excluded from `tests/`**: rejected — would have permanently carved out an exception for exactly the checks most likely to catch real regressions in the test suite's own idioms, on the basis of one-time cleanup cost rather than any disagreement with what they're flagging.
- **Extend all 6 checks to `tests/` and reconcile every violation by fixing the test code**: adopted. No blanket `# pytriage: ignore` suppression to preserve the `tree =`/`check =` arrange-step idiom — it was collapsed into the call it fed (e.g. `ForbidVarsCheck().check(Path("test.py"), ast.parse(source), source)`), and the 4 `result` names were renamed to what they held (`fix_applied`, `filtered_files`, `fixed_content`).

## Consequences

- `.pre-commit-config.yaml`'s `ruff-extra-rules` hook now runs against `files: ^(src|tests)/` with `exclude: ^tests/fixtures/`, `args: [--fix]` unchanged.
- 198 of 280 violations were resolved by `--fix` (run to convergence — a single pass left some fixable violations unapplied, a cascading-fix quirk, not a correctness issue); the remaining 82 were fixed by hand across `tests/test_forbid_vars.py`, `test_orchestrator.py`, `test_redundant_assignment.py`, `test_misplaced_comment.py`, `test_excessive_blank_lines.py`, `test_redundant_super_init.py`, and `test_validate_function_name.py`. All 557 tests still pass.
- No genuine false positives turned up — every violation was a legitimate instance of what its check flags, including under the added stress of the check suite analyzing its own test suite (`redundant_assignment` and `forbid_vars` analyzing code that constructs `RedundantAssignmentCheck`/`ForbidVarsCheck` instances).
- Two real autofix gaps in `redundant_assignment` surfaced during the manual pass, unrelated to whether `tests/` specifically is linted: a `[FIXABLE]` label that `--fix` silently never applies for some long use-lines (`should_autofix()`'s line-length estimate disagreeing with `_can_safely_inline()`'s real one), and `IMMEDIATE_SINGLE_USE` never allowing a `Call` RHS even for trivial zero-arg constructors (unlike the more permissive non-immediate `SINGLE_USE` pattern). Tracked in [issue #22](https://github.com/alessio-locatelli/ruff-extra-rules/issues/22) rather than fixed inline — improving autofix precision is a distinct concern from scoping which directories get linted.
- Tracked as `docs/work-items/archived/10-lint-tests-directory.md`.
