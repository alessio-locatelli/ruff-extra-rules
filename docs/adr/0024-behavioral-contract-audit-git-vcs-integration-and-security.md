# Behavioral contract audit: Git/VCS integration & security (ch. 22, 23)

`docs/behavioral_contract.md` chapters 22 (Git and VCS Integration) and 23 (Security) were audited against every `git` subprocess call and every temp/cache-file write. No `eval`/`exec`/dynamic import exists anywhere, no subprocess call goes through a shell, and every temp file `atomic_write_text()` writes already uses an unpredictable `tempfile.mkstemp()` name. Full findings are in `docs/audits/0013-behavioral-contract-audit-git-vcs-integration-and-security.md`.

## Decision

Plain `git grep <pattern> -- <path>` only searches tracked files, even when the path is named explicitly on the command line — an untracked or `.gitignore`d file passed directly makes `git grep` exit with empty stdout and stderr, indistinguishable from a genuine non-match, so `git_grep_filter()` silently dropped it from every check's candidate list. This mattered most for the direct-CLI workflow (`AGENTS.md`'s documented dev usage), the exact moment a file is most likely to not yet be staged. `git_grep_filter()`'s invocation now always adds `--untracked --no-exclude-standard`, so it searches the working-tree content of every explicitly-named path regardless of index/ignore status. `_list_python_files_in_dir()`'s own `.gitignore`-aware `git ls-files` is deliberately left unchanged — that's a directory scan where excluding untracked scratch files is the intentional, already-documented behavior (ADR 0015), not the same gap.

**Amended by `docs/adr/0027-behavioral-contract-audit-synthesis.md`:** viewed together with the fix above rather than in this chapter's own isolation, the resulting asymmetry — an untracked file is always searched when named explicitly, but silently excluded from a directory scan with no diagnostic — reproduces the identical false-clean-result failure mode this ADR fixed for `git_grep_filter()`. `docs/adr/0027` revises this paragraph's "not the same gap" conclusion and tracks the fix as [issue #67](https://github.com/alessio-locatelli/ruff-extra-rules/issues/67).

`_cache.py`'s `_write_cache()` wrote through a fixed, fully-predictable temp-file name via a plain `open(..., "w")`, which follows a symlink rather than refusing to — a classic TOCTOU: a symlink pre-planted at that path gets its target overwritten with cache content on the next write. It now uses `tempfile.mkstemp(dir=cache_file.parent, ...)`, the same pattern `atomic_write_text()` already established, so a pre-planted symlink at the old name is simply never opened. The equivalent predictable-path exposure in `_ensure_cache_dir()`'s one-time `CACHEDIR.TAG` write and in `_locked()`'s lock file were both judged not worth hardening: the former writes only a fixed, public, non-sensitive banner; the latter's name must be predictable and shared by design (that's how two processes agree which file to lock), and nothing is ever written through it.

## Consequences

- `git_grep_filter()` always passes `--untracked --no-exclude-standard`; the command still only ever searches the caller's own explicit pathspec, so no new files are examined beyond what was already going to be checked.
- `_write_cache()`'s temp files are now named `.{cache_file.name}.<random>.tmp` instead of a hash-derived fixed name, closing the symlink-following write. No behavior change for a well-behaved run — this only changes what an attacker who can already write to `.cache/pre_commit_hooks/` can achieve.
