# Behavioral contract audit: file discovery, path handling & incremental scope (ch. 12, 13)

`docs/behavioral_contract.md` chapters **12** (Incremental Execution) and **13** (Filesystem and Path Handling) were checked line-by-line against `_prefilter.py`'s `git grep`/`git ls-files` subprocess calls, `ast_checks/__init__.py`'s CLI argument handling and `CheckOrchestrator`, and `_cache.py`'s path-based cache keys, following on from `0011`–`0014`'s audits of chapters 1, 3, 4, 5, 6, 11, 20, 24, 25, and 30, per issue #33.

Most items were already satisfied: every path used throughout this pipeline is passed as a plain string argument to `subprocess.run()` (never through a shell — grepped for `shell=True`/`os.system`/`os.popen`, none found), so paths containing spaces, Unicode, or shell-special characters, and shell-injection risk, are all non-issues, verified empirically with a path literally containing `;&&|$(...)`. File identity is always established via `Path.resolve()` (`_cache.py`'s `_get_cache_path`, `_prefilter.py`'s `input_map`), so a relative and an absolute spelling of the same file correctly share one cache entry, and two distinct files (verified with same-directory differently-cased names, since this project assumes a case-sensitive Linux filesystem per `AGENTS.md`) are never conflated. A missing file, a broken symlink, and a permission-denied file each raise a subclass of `OSError` from `read_source_with_encoding()`/`Path.stat()`, already caught by `0011`'s `unprocessable_files` reporting — none of them crash the run. Every write goes through `atomic_write_text()`/`_write_cache()`'s temp-file-then-`replace()` pattern, already documented (that docstring) to act on a symlink's resolved target rather than the symlink's own directory entry, and no code anywhere assumes a filesystem operation is atomic beyond what POSIX `rename()` actually guarantees. The cache's mtime fast-path is verified-then-trusted, not trusted alone: a content hash is (re-)computed and compared whenever mtime or size differs, so it doesn't "rely solely on timestamps" in the sense ch. 13 means. No check performs cross-file analysis (`redundant_super_init.py`'s own inheritance-chain lookup, the closest candidate, only ever resolves classes from the _same_ file's own AST) — grepped confirmed no check reads any file `_check_file()` didn't hand it — so every ch. 12 item about invalidating or diverging incremental-vs-full results was, by construction, already impossible to violate; `docs/adding-a-check.md` now states that invariant explicitly (the one genuine documentation gap: nothing previously said this in writing) so a future check can't quietly break it.

## Gaps found and fixed

- **`_prefilter.py`'s `git_grep_filter()` silently dropped a file `git grep` couldn't process from every check's candidate list, with no trace it was ever skipped** (ch. 13: "MUST NOT silently skip inaccessible files", "MUST report inaccessible files clearly", "MUST handle permission-denied files without crashing the entire run"; ch. 12: "MUST handle files added, deleted, renamed, or moved between discovery and processing"). Reproduced two distinct ways inside a real git repo (the earlier `test_prefilter.py`/`test_orchestrator.py` tests only ever exercised the _not-a-git-repo_ Python-fallback path, since `pytest`'s `tmp_path` isn't part of any repo — so this exact branch had zero coverage before now): a permission-denied tracked file makes `git grep` print `error: failed to stat '<path>': Permission denied` to stderr while still exiting 0 (if another requested file matched) or 1 (if none did) — exit codes the old code treated as unconditionally trustworthy success/no-match signals, never once inspecting stderr; a file that vanished between the caller's file list being built and this call (e.g. deleted after pre-commit computed the changed-file list but before this hook actually ran) makes `git grep` exit 1 with _empty_ stdout and _empty_ stderr — indistinguishable, at the process-output level, from a genuine non-match. Since 5 of this project's 6 checks declare a prefilter pattern (only `excessive-blank-lines` doesn't), and `main()` loads every check by default, this was the code path a default, no-flags invocation actually takes for nearly every file — not a rare corner case. Fixed by (1) computing `os.access(fp, os.R_OK)` for every input path up front and always carrying anything not confirmed-readable through as a candidate regardless of what `git grep` reports for it, closing the "vanished, silent" case; and (2) only trusting `git grep`'s stdout when its stderr is also empty, falling back to `_python_fallback_filter()` otherwise — which already reads each file itself and, per its own pre-existing contract, includes a file it can't read "so the hook can handle the error" rather than dropping it. Both paths defer the actual diagnosis to the existing `_check_file()`/`_read_source()` reporting `0011` already built, rather than teaching the prefilter layer to explain _why_ a file is unreadable. Regression tests: `test_git_grep_filter_includes_permission_denied_tracked_file`, `test_git_grep_filter_includes_file_deleted_since_discovery` (`tests/test_prefilter.py`, both using a real `git` subprocess, no mocking) and `test_main_permission_denied_file_returns_one_inside_git_repo` (`tests/test_orchestrator.py`), the last proving the fix at the level a user actually experiences it: `main()` on a permission-denied file inside a real git repo now exits 1 and prints `<path>: error: could not be read or parsed; file skipped` instead of silently exiting 0. All three fail against the pre-fix code (verified by temporarily reverting just `_prefilter.py` and re-running them).

- **A directory CLI argument (e.g. `ruff-extra-rules src/`, the exact form `AGENTS.md`'s own "Running checks directly" dev docs use) reached `CheckOrchestrator.process_files()` unexpanded and silently checked nothing** (ch. 12: "MUST process only the requested scope" — a directory _is_ the requested scope here, and processing zero files under it is a failure to honor that, not a narrowing of it). `pre-commit`/`prek` itself never passes a directory (`.pre-commit-hooks.yaml`'s `types: [python]` hook contract always expands to individual files before invoking any hook), so this only affects direct CLI use — but that's a real, documented, currently-broken usage path, not a hypothetical one. Root cause, found while investigating the gap above: `git grep`'s own directory-pathspec support makes it recurse and report matches for files _inside_ the given directory, but `git_grep_filter`'s `input_map` only ever contains the literal (unresolved-to-children) directory path itself, so every one of those matches was already being discarded by the pre-existing "skip a git result that doesn't resolve back to a requested input path" defensive check (see `test_git_grep_filter_skips_unresolvable_git_paths`) — `candidate_files` ended up empty, `process_files()` returned `{}`, and `main()` exited 0 having read zero bytes. Fixed by adding `expand_directories()`, called in `main()` right after the "no files to check" guard and before `--exclude` filtering: any argument that's a directory is replaced with the `.py` files under it, resolved to absolute paths (`git ls-files` reports paths relative to whatever `-C` directory it's run against, which would otherwise make a directory argument's expansion inconsistent with a plain file argument passed through in its original relative-or-absolute spelling — ch. 13: "MUST handle relative and absolute paths consistently" — so both the git and non-git branches resolve to the same absolute form). Prefers `git ls-files` (tracked, `.gitignore`-aware, consistent with `_prefilter.py`'s own git-based discovery — a build artifact or `.venv/` living under the given directory is never swept in) and falls back to a plain recursive glob outside a git repo or when git itself is unavailable, mirroring `_prefilter.py`'s own established git-then-Python-fallback shape. A non-directory argument (an ordinary file, or a path that doesn't exist at all) is left untouched, so the existing unreadable/unprocessable-file reporting still applies to it exactly as before. Regression tests: `test_expand_directories_leaves_plain_files_untouched`, `test_expand_directories_globs_python_files_outside_a_git_repo`, `test_expand_directories_uses_git_ls_files_inside_a_git_repo` (the last also proving `.gitignore`/tracked-only scoping: an untracked file under the directory is correctly excluded), and `test_main_directory_argument_checks_files_inside_it` (`tests/test_orchestrator.py`) — proving the fix at the CLI level: `main([str(a_directory)])` on a directory containing a real violation now exits 1 and reports it, where it previously exited 0 silently. Manually re-verified against the exact `AGENTS.md`-documented command (`... --select=forbid-vars,validate-function-name src/`), which now reports (and, once fixed, stops reporting) a real `forbid-vars` violation the new `_list_python_files_in_dir()` helper's own first draft introduced (a variable named `result` — this project dogfoods its own `forbid-vars` check).

- **`_list_python_files_in_dir()`'s `git ls-files` branch could list a `.py` file that no longer exists on disk** (ch. 12: "MUST avoid relying on stale file lists when the user explicitly requests a current filesystem state"). Found during code review of the fix above: `git ls-files` reports the _index_, not the working tree — a tracked file removed with a plain `rm` (rather than `git rm`) still shows up in its output. Unlike an individually-named file argument (which `git_grep_filter`'s own fix above always still surfaces, since the user explicitly asked about that specific path and its removal is itself news worth reporting), a directory scan never named that file — it's asking "what's currently under here", so a stale index entry for a file that's actually gone would have surfaced a misleading `<path>: error: could not be read or parsed; file skipped` for a file the user never referenced, on an otherwise-clean directory. Fixed by filtering the `git ls-files` output to entries that still `Path.exists()` before returning them; the non-git `rglob()` fallback branch needed no change, since a filesystem walk can't list a file that isn't there. Regression test: `test_expand_directories_skips_tracked_file_deleted_from_working_tree` (`tests/test_orchestrator.py`), which fails against the pre-fix code (verified by temporarily reverting just the `.exists()` filter and re-running it).

## Items judged not applicable, with rationale

- **Ch. 13, "MUST handle filesystems with unusual timestamp resolution safely"**: `_cache.py`'s mtime-fast-path/hash-fallback design (see above) means a coarse-resolution filesystem can only produce a stale cache hit in the narrow window where a file is rewritten with byte-identical size within the same timestamp tick as its previous write — the same accepted tradeoff `mypy`'s own incremental cache makes, on `st_mtime_ns` (nanosecond) resolution rather than whole-second, and scoped to a project that (per `AGENTS.md`) assumes Linux/WSL only, where ext4/xfs/btrfs all give nanosecond-resolution timestamps natively. Not fixed: always re-hashing regardless of mtime would defeat the fast path's entire purpose to close a theoretical gap this project's own supported-platform assumption already narrows to near-zero.
- **Ch. 13, "MUST handle network filesystems as gracefully as possible"**: `CacheManager._locked()`'s `fcntl.flock()` (hardened for timeout by `0013`) is a well-known unreliable primitive over pre-NFSv4 mounts — it can silently fail to provide real mutual exclusion. Not fixed: this is a single-maintainer local hobby project with no stated network-filesystem support, `.cache/pre_commit_hooks/` is ordinary project-local state, and even a worst-case lock failure here only risks the same lost-update race the lock exists to reduce — a stale cache entry a later run overwrites again — never data loss or a crash.

## Consequences

- `_prefilter.py`'s `git_grep_filter()` now does one `os.access()` call per input path per pattern (cheap — a single `stat()`-class syscall, not a read) before ever invoking `git grep`; negligible next to the subprocess spawn it already pays per pattern.
- `git_grep_filter()`'s fallback trigger widened: previously only a non-{0,1} returncode or a raised exception fell back to `_python_fallback_filter()`; now a 0 or 1 returncode with non-empty stderr does too, which means the (slower, one-`open()`-per-file) Python path runs somewhat more often on a git repo with any per-file `git grep` error — an intentional, safety-over-speed tradeoff, matching this project's own existing "fall back to Python on any real git error" shape.
- `ast_checks/__init__.py` gains two new module-level functions, `expand_directories()` (public, alongside `filter_excluded_files()`) and `_list_python_files_in_dir()` (private helper), and a new `subprocess` import; `main()` now expands directory arguments before `--exclude` filtering, so an exclude glob written against an individual file path (e.g. `migrations/*.py`) still applies correctly to files that came from a directory argument.
