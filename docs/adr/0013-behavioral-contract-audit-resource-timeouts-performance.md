# Behavioral contract audit: resource usage, timeouts & performance (ch. 24, 25, 30)

`docs/behavioral_contract.md` chapters **24** (Resource Usage), **25** (Timeouts and Hanging Operations), and **30** (Performance) were checked line-by-line against `CacheManager` (`_cache.py`), `_prefilter.py`'s `git grep` subprocess call, and every check's own AST traversal, following on from `0011`/`0012`'s audits of chapters 1/3/4/5/6/20.

Most items were already satisfied: files are read and parsed one at a time rather than loaded wholesale (ch. 24: "avoid loading an unbounded number of entire files into memory"), every temp file (`atomic_write_text()`, `_write_cache()`) is already cleaned up on both the success and failure paths, the `git grep` subprocess already carries a `timeout=30` with a tested fallback path, there is no parallel-worker code in this repository for a worker to hang or crash in, and `0012`'s per-check `except Exception` isolation already turns _any_ unexpected exception — including a `RecursionError`, verified below — into a reported, exit-code-1 `rule_failures` entry rather than a silent or corrupted result.

## Gaps found and fixed

- **`CacheManager._locked()`'s `fcntl.flock(lock_fp, fcntl.LOCK_EX)` blocked indefinitely, with no timeout** (ch. 25: "MUST avoid indefinite hangs", "MUST define timeout behavior for operations that can block indefinitely", "MUST NOT leave the user with an apparently hung process and no way to determine what is happening"). The lock exists specifically because "multiple hook processes (e.g. under prek's parallel execution) can target the same per-file cache blob... at the same time" (the method's own docstring) — a real, acknowledged multi-process scenario, not a hypothetical one. A crashed peer releases its `flock` automatically when its file descriptor closes (even on `SIGKILL`), so the only way to actually hang here is a peer that's still genuinely running slowly (e.g. a loaded machine or a slow filesystem) — but the old code had no escape from that case at all. Fixed by polling a non-blocking `LOCK_EX | LOCK_NB` attempt against a 10-second deadline (`_LOCK_TIMEOUT_SECONDS`) instead of a single blocking call, raising `TimeoutError` if the deadline passes. `TimeoutError` is a builtin subclass of `OSError`, so both `get_cached_result`'s and `set_cached_result`'s existing `except OSError` clauses already treat it exactly like any other cache failure — log a warning, degrade to an uncached result (ch. 11: "MUST gracefully degrade to uncached execution when caching is unavailable") — with no new exception handling needed at either call site. Regression tests: `test_get_cached_result_degrades_to_cache_miss_when_lock_times_out`, `test_set_cached_result_does_not_hang_when_lock_times_out` in `tests/test_cache.py`, both holding a real OS-level `flock` from a second file descriptor on the same lock path (the same contention a second process would cause) and asserting the call returns well under the real 10-second default instead of hanging.

- **`validate_function_name/analysis.py`'s `attach_parents()` was hand-written recursion with no depth bound, run unconditionally over the _entire_ tree of every file the check processes** (ch. 24: "MUST avoid pathological behavior caused by deeply nested source code where possible", "MUST handle recursion limits safely"). Verified empirically: `ast.parse()` itself still accepts `"x = " + "not " * 1000 + "True"` as ordinary valid Python (this project's own precondition — see `AGENTS.md` — is that a file reaching this hook already parsed once via `check-ast`), but the old recursive `attach_parents()` raised `RecursionError` on that same source at exactly that depth, well below what `ast.parse()` itself tolerates (parsing kept succeeding up to at least 5000x nesting before hitting an unrelated CPython parser-stack `MemoryError` around 20000x, a pre-existing, out-of-scope limit `ast.parse()` itself would already hit before this hook ever sees the file). Since `collect_suggestions()` calls `attach_parents(tree)` unconditionally for every file matching the `def get_` prefilter, this was a plain crash on an ordinary (if unusual-looking), non-adversarial construct — not a contrived edge case. Fixed by rewriting `attach_parents()` to use an explicit stack instead of recursive calls; behavior is unchanged (same parent pointers, same traversal), only the risk of blowing the call stack is removed. Verified up to 5000x nesting with no error. Regression test: `test_attach_parents_handles_deeply_nested_source_without_recursion_error` in `tests/validate_function_name/test_analysis.py`.

- **`ForbiddenNameVisitor.visit_Assign`/`visit_AnnAssign` (`forbid_vars.py`) and `VariableTracker._get_source_segment` (`redundant_assignment/analysis.py`) called `ast.get_source_segment(self.source, node)` once per assignment node**, and `ast.get_source_segment` re-splits its _entire_ `source` argument into lines internally on every call (see its stdlib implementation) — turning a linear traversal into O(assignments × source size) (ch. 24: "MUST avoid quadratic or worse behavior on common large inputs unless explicitly justified"; ch. 30: "MUST avoid pathological performance on common Python constructs"). This wasn't a synthetic worst case: a single 2000-function generated-looking file (e.g. a generated API client or ORM models module — an ordinary, common shape, not adversarial) took **3.5s** in `forbid-vars` and **2.6s** in `redundant-assignment` alone, more than the entire rest of this repository's own 110 tracked `.py` files combined (**~2.1s** cold, running all six checks). Fixed by adding `fast_get_source_segment()` to `_base.py` — same result as `ast.get_source_segment` for a single-line node, computed from a precomputed line list instead of re-splitting on every call, falling back to the real `ast.get_source_segment` for the rare node spanning multiple lines. Measured effect: `forbid-vars` 3.5s → ~0.12s (~29x) and `redundant-assignment` 2.6s → ~0.32s (~8x) on the same 2000-function file; the whole-repo cold run improved from ~2.1s to ~0.5s. Regression tests: `test_fast_get_source_segment_matches_ast_get_source_segment` (parametrized: single-line, unicode-before-node, multiline-parenthesized, single-line-subscript, no-trailing-newline, form-feed-inside-single-line-node) and `test_fast_get_source_segment_returns_none_without_end_position` in `tests/test_base.py`, proving byte-for-byte equivalence with `ast.get_source_segment` rather than merely asserting it's faster; the existing `forbid_vars`/`redundant_assignment` suites (309 tests) already re-ran unchanged and green, since violation counts and content are identical before and after.

  The first version of this fix reused each class's existing `self.source_lines = source.splitlines()` (already computed for other position lookups) as the precomputed line list, and a review pass caught that this was wrong: `str.splitlines()` additionally splits on form feed and several other Unicode line-separator characters that Python's own tokenizer treats as ordinary intra-line content, not a line boundary — all legal, if unusual, inside otherwise ordinary Python source (e.g. a string literal containing a raw form-feed byte). A node fully on one _parser_ line but straddling one of those characters would hit the fast path (`node.end_lineno == node.lineno`) yet index into the wrong, truncated `str.splitlines()` entry — a real correctness regression relative to the `ast.get_source_segment` call it replaced, not merely a performance-neutral refactor. Fixed by adding `split_lines_like_ast()` to `_base.py` — splits only on `\r\n`/`\n`/`\r`, matching `ast`'s own private `_splitlines_no_ff` line-boundary rule (reimplemented rather than depending on that private function directly) — and giving both classes a _second_, dedicated `self._ast_lines` field for this purpose, computed once alongside (not replacing) the existing `self.source_lines`, which other pre-existing lookups in the same classes still use unchanged. Regression tests: the `form-feed-inside-single-line-node` case above, and `test_split_lines_like_ast_matches_ast_own_line_numbering` (parametrized: LF, CRLF, CR, no-trailing-newline, form-feed-is-not-a-boundary, empty) proving agreement with `ast`'s own private splitter.

## Items judged not applicable, with rationale

- **Ch. 24, "MUST avoid unbounded cache growth"**: `.cache/pre_commit_hooks/` never prunes an entry for a file that's since been deleted or renamed — growth is bounded only by the total number of distinct file paths ever checked over the project's lifetime, not by the current working tree. Not fixed: this matches the same convention `mypy` (`.mypy_cache`) and `ruff` (`.ruff_cache`) both ship with — neither auto-prunes stale per-file entries either — and this project's own `CACHEDIR.TAG` already self-documents the accepted maintenance path ("It is safe to delete this directory to clear the cache", per the [Cache Directory Tagging spec](https://bford.info/cachedir/)). Building automatic staleness tracking (diffing every cache entry against the current working tree on each run) would add ongoing directory-scan overhead to every invocation to solve a problem a single `rm -rf .cache` already solves for a single-maintainer local hook.
- **Ch. 24, "MUST avoid unbounded diagnostic accumulation where output can be streamed or bounded"**: `all_violations` accumulates every violation for the whole run before `main()` prints anything, rather than streaming per file. Not changed: content is bounded by the number of actual violations found, not by file content or file count (files themselves are still processed and released one at a time — see the "avoid loading an unbounded number of entire files" item above, which _is_ satisfied) — the same "accumulate diagnostics, print at the end" shape most CLI linters use for a non-streaming output format.
- **Ch. 24, "MUST provide reasonable behavior for very large files" / "MUST define behavior when a file exceeds supported size limits"**: no explicit size cap exists anywhere in this pipeline. Not fixed: after the quadratic `get_source_segment` fix above, a 2000-function/120KB single file now completes in under a second across all six checks — no evidence of pathological behavior on a large-but-ordinary file remains, and this project's own precondition (`AGENTS.md`) is that every file it processes already passed `ruff check`/`check-ast` in the same local pre-commit run, which would itself already be slow on a truly pathological file before this hook ever saw it. An artificial cap with no concrete failure mode behind it would be exactly the kind of scenario-that-can't-happen defensive code this project's own guidelines ask not to add.
- **Ch. 24, remaining hand-rolled and `ast.NodeVisitor`-based recursion** (`_scope.py`'s `_walrus_targets`/`iter_within_scope`, `redundant_assignment/analysis.py`'s `_call_precedes_target`, `redundant_super_init.py`'s `_parent_accepts_args`, `validate_function_name/autofix.py`'s `_count_nesting_depth`, `validate_function_name/analysis.py`'s `_get_base_name`, and every check built on `ast.NodeVisitor` — `ForbiddenNameVisitor`, `SuperInitChecker`, `VariableTracker`, `NonDeterministicCallDetector`, `_ReferenceCollector`): all share the same unbounded-recursion shape as `attach_parents()`, but each is bounded by a narrower dimension than whole-file AST depth — a single scope's own statement/expression nesting, a class's inheritance-chain depth, an attribute-chain length, or (for `_count_nesting_depth`) a function already known to be under 20 lines before it's ever called. `ast.NodeVisitor.generic_visit()` itself (CPython's own stdlib implementation) is _inherently_ recursive with no depth bound; replacing it across five checks' core traversal engines would be a fundamental architecture change, not an audit-scoped fix. Verified empirically instead that this is already handled safely in the sense chapter 24 asks for: running the four `NodeVisitor`-based checks (`forbid-vars`, `redundant-super-init`, `validate-function-name`, `redundant-assignment`) against the same 1000×-nested-`not` source that broke `attach_parents()` does raise `RecursionError` in each — but `0012`'s existing per-check `try/except Exception` in `_check_file()` already catches it, logs it, records `(filepath, check_id)` in `rule_failures`, leaves every other check and file completely unaffected, and forces `main()`'s exit code to `1` with a clear message — never a silent or corrupted result. Only `attach_parents()` (unconditional, whole-file, and trivially safe to convert with zero behavior change) crossed the bar for an actual code fix; the rest are documented here as an accepted, already-isolated residual risk rather than rewritten.
- **Ch. 25, "MUST report which operation timed out where possible" for `_prefilter.py`'s `git grep` subprocess**: the single `except (subprocess.SubprocessError, FileNotFoundError, subprocess.TimeoutExpired)` clause logs via `logger.exception("git grep failed")` for every cause in that tuple alike, without a message that names "timeout" specifically. Not changed: `logger.exception()` includes the caught exception's own type and message in the log record, and `subprocess.TimeoutExpired`'s own message already states the command and the 30-second timeout that elapsed — the information chapter 25 asks for is already present in the existing log line, just via the exception's own `repr`, not a hand-written distinct string.
- **Ch. 25, TTY/signal/process-group assumptions**: this codebase contains no `signal` module usage, no TTY/`isatty()` checks, and spawns no long-lived subprocess or worker beyond the single bounded, already-timed-out `git grep` call — grepped and confirmed empty. Nothing here to assume incorrectly about a terminal or signal behaving differently across environments.
- **Ch. 30, "MUST avoid making startup overhead dominate execution for small inputs"**: `CheckOrchestrator.__init__` re-hashes this package's own `src/` tree on every invocation via `compute_tree_hash()`, regardless of whether anything changed. Not a gap: `compute_tree_hash()`'s own docstring already documents this as measured (~0.2ms for this repo's own tree), negligible next to interpreter startup — this was a deliberate, already-justified tradeoff from `0005`, not something this audit needed to revisit.
- **Ch. 30, "MUST measure performance regressions where performance is a project requirement"**: no `pytest-benchmark`-style infrastructure was added — the concrete numbers cited above (whole-repo cold run, single large file, per-check isolation) were gathered with a throwaway timing script for this audit, not committed as an automated benchmark suite, matching the acceptance criteria's own allowance for "a benchmark note for performance-only findings" rather than a new dependency and CI-less flaky wall-clock assertions for a single-maintainer local repo with no CI to run them in.

## Consequences

- `CacheManager._locked()` can now raise `TimeoutError` (a new, documented possibility) instead of blocking forever; every existing caller already handles it via its existing `except OSError`, so no call site needed to change.
- `_base.py` gains two new public helpers, `fast_get_source_segment()` and `split_lines_like_ast()`, alongside `byte_col_to_char_col()`; `ast` is now a real (not `TYPE_CHECKING`-only) import there, since the new helpers call `ast.get_source_segment()` for the multi-line fallback.
- `validate_function_name/analysis.py`'s `attach_parents()` keeps its exact signature and behavior (same parent pointers on the same nodes) — only its internal traversal mechanism changed, from recursive to an explicit stack.
- `forbid_vars.py`'s `ForbiddenNameVisitor` and `redundant_assignment/analysis.py`'s `VariableTracker` each gain a second precomputed field, `self._ast_lines` (alongside the existing `self.source_lines`, which every other pre-existing call site in both classes still uses unchanged), and produce byte-identical violation output to before (proven by the existing 309-test suite re-running unchanged), just faster on files with many assignments.
