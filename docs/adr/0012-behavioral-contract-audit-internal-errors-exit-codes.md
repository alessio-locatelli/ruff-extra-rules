# Behavioral contract audit: internal error isolation & CLI exit codes (ch. 5, 6)

`docs/behavioral_contract.md` chapters **5** (Internal Errors and Failure Isolation) and **6** (CLI Exit Codes) were checked line-by-line against `CheckOrchestrator` (`ast_checks/__init__.py`) and `main()`'s own reporting/exit-code logic, following on from `0011`'s audit of chapters 1/3/4/20.

Most items were already satisfied: `CheckOrchestrator._check_file()` already isolates a check's `check()` exception per-file-per-check (one broken check can't stop another check, or another file, from being processed), `_apply_fixes()` already isolates a check's `fix()` failure the same way, `atomic_write_text()` already guarantees a rejected fix leaves the file untouched, and `main()` already turns every reported violation, and every unreadable/unparseable file (`0011`), into a non-zero exit code with a file-and-line diagnostic.

## Gaps found and fixed

- **A check whose `check()` raised on every file it saw made the whole run look completely clean.** `_check_file()`'s per-check `try/except Exception` already logged the failure and let other checks keep running (ch. 5: "prevent one broken rule from corrupting the results of unrelated rules" was already true) — but nothing distinguished "this check crashed and found nothing" from "this check ran and genuinely found nothing." If it was the only check enabled (or the only one that would have flagged the file), the file got zero entries in `all_violations`, `main()` never printed anything for it, and the run exited `0` (ch. 5: "MUST NOT hide internal errors behind a successful exit status"; ch. 6: "MUST NOT return success merely because the process did not crash", "MUST return a non-zero exit code when a fatal error prevents reliable analysis"). Worse, in non-fix-mode this incomplete result was then cached, so a later cache-hit run would keep serving the crash's "clean" result until the tree hash changed, without ever retrying the check. Fixed by adding `CheckOrchestrator.rule_failures: list[tuple[str, str]]` (filepath, check_id pairs), populated whenever `_check_file()`'s per-check loop catches an exception and reset at the start of every `process_files()` call — mirroring `0011`'s `unprocessable_files`. `process_files()` now skips caching for any file a check crashed on, and `main()` prints one line per (file, check) failure and sets `exit_code = 1`. Regression tests: `test_process_files_check_exception_records_rule_failure`, `test_process_files_rule_failure_is_not_cached`, `test_main_check_crash_returns_one_and_reports_check_and_file` in `tests/test_orchestrator.py`.
- **A check's `fix()` raising anything other than `FixValidationError` was indistinguishable, in the printed report, from a violation that simply hadn't been fixed yet.** `_apply_fixes()`'s outer `except Exception` already caught it (logged, file left as whatever state `fix()` left it in, other checks unaffected — ch. 5 isolation was fine), but the violation's `fix_data` was never updated, so `main()` reported it as plain `[FIXABLE]` with "Run with --fix to inline automatically" — actively misleading, since `--fix` had already been attempted and had already crashed the same way (ch. 5: "MUST report the affected rule and file when a rule implementation fails", "MUST distinguish user errors from tool bugs where possible"; ch. 6: "MUST distinguish, where useful, between ... violations fixed ... internal tool failure"). Fixed by adding a second `except Exception` around specifically the `check.fix()` call (after the existing `except FixValidationError`), calling a new `mark_fix_errored()`/`is_fix_errored()` pair (`_base.py`, mirroring `mark_fix_rejected()`/`is_fix_rejected()`) and reporting it as `[FIX ERRORED]` with the same "this is a bug, please report it" hint `[FIX REJECTED]` already used, instead of the ordinary "run with --fix" suggestion. Scoped deliberately narrow: only the `fix()` call itself is wrapped this way, not the surrounding re-read/recompute/post-fix-recheck calls in the same method, which remain covered by the outer catch-all (see below).

  Three refinements came out of review, all about not letting the new `[FIX ERRORED]` marking itself become a new way to misreport state:
  - A check that writes more than once per `fix()` call (looping over violations individually, like `validate_function_name`) can have already committed some of `fresh_violations` before a later one's write raised — naively marking the whole batch errored would misreport an already-applied fix as "not applied." Fixed by extracting `CheckOrchestrator._mark_resolved_and_get_still_present()` (re-reads the file, re-runs `check.check()`, calls `mark_fixed()` on anything no longer present, returns the still-present set) and reusing it in both the existing success path and the new exception path, so only violations still actually present after the crash get `[FIX ERRORED]`.
  - `validate_function_name`'s own `fix()` (`validate_function_name/__init__.py`) already had its own internal `except Exception: logger_check.exception(...)` around each individual `apply_fix()` call, predating this audit — it caught and logged an unexpected exception per-violation without ever letting it propagate, so `CheckOrchestrator`'s new handling above was structurally unreachable for the one check that actually writes more than once per call. Fixed at the source: that `except Exception` branch now also calls `mark_fix_errored(violation)`, mirroring the `except FixValidationError` branch immediately above it in the same loop.
  - The opposite edge case: a `fix()` that commits every requested edit and then raises afterwards (e.g. during unrelated cleanup) leaves nothing left to mark `[FIX ERRORED]` — the post-fix re-check correctly marks every one of `fresh_violations` `[FIXED]`, so the exception itself would otherwise be visible only in the debug log, behind what reads as a fully successful run (ch. 5: "MUST NOT hide internal errors behind a successful exit status" applies just as much here as to the check()-crash gap above). Fixed by having this `except Exception` handler always append `(filepath, check.check_id)` to `rule_failures` — the same mechanism the first gap above introduced — regardless of how many of `fresh_violations` end up resolved, so `main()` always prints the failure and forces `exit_code = 1`, never silently behind an all-`[FIXED]` report.

  Regression tests: `test_apply_fixes_marks_violation_errored_when_fix_raises_unexpectedly`, `test_apply_fixes_marks_already_resolved_violation_fixed_not_errored`, `test_apply_fixes_marks_errored_violation_of_a_multi_write_check_when_apply_fix_raises`, `test_apply_fixes_records_rule_failure_when_fix_raises_after_resolving_everything`, `test_main_fix_flag_reports_errored_fix`, `test_main_reports_rule_failure_when_fix_raises_after_resolving_everything` in `tests/test_orchestrator.py`; `test_mark_fix_errored`/`test_is_fix_errored_false_when_only_marked_rejected` in `tests/test_base.py`; `test_fix_marks_violation_errored_and_continues_when_apply_fix_raises` in `tests/validate_function_name/test_check.py`.

Both gaps were about surfacing an already-isolated internal failure to the user and the exit code, not about making failure isolation itself more aggressive — the per-check, per-file `try/except` boundaries were already correct.

## Items judged not applicable, with rationale

- **Ch. 6, "MUST distinguish ... violations fixed" via a distinct exit code**: a `--fix` run that resolves every violation still returns `1`, the same as an unfixed run — not a gap. This project's hook id (`ruff-extra-rules`) is a pre-commit/prek hook first (see `CONTEXT.md`), and a hook that modifies files is expected to fail the run so pre-commit stops the commit for the user to review and re-stage the diff, the same convention `black`/`ruff-format`/`end-of-file-fixer` all follow, and the same effect `ruff check --fix --exit-non-zero-on-fix` opts into deliberately when driven from a pre-commit hook. `--exit-zero`/`--exit-non-zero-on-fix`-style flags were already explicitly deferred by `0008`. The distinction chapter 6 asks for ("where useful") is already made in the printed report via the `[FIXED]` tag, just not via the exit code.
- **Ch. 5, "MUST NOT silently ignore unexpected exceptions" for `load_checks()`'s own `check_class()`/`check_class(**args)` construction failures**: already caught, logged, and skipped (`test_load_checks_skips_check_whose_init_raises`, `test_load_checks_skips_check_when_custom_args_raise`), but not reflected in `main()`'s exit code the way `rule_failures` now is for a per-file `check()` crash. Not extended to also affect the exit code: `ALL_CHECKS` is a fixed, first-party list with no plugin mechanism (`ast_checks/__init__.py`'s own comment on `ALL_CHECKS`), none of the 6 shipped checks defines a custom `__init__` that could plausibly raise, and this project's own local dev-check sequence (`AGENTS.md`) would catch a check that broke its own constructor before it ever reached a user. Revisit if a plugin mechanism is ever added.
- **Ch. 5, items 11/12 ("MUST define whether fixes are committed per file, per batch, or per entire invocation", "MUST ensure that the chosen commit boundary is safe and deterministic")**: already true but previously undocumented as such. Fixes are committed per (check, file): each check's own `fix()` call writes and atomically replaces the file (`atomic_write_text()`) immediately, in `self.checks`' fixed order (`ALL_CHECKS`, filtered by `--select`/`--ignore`) — never batched across checks or across the whole invocation. This is deterministic (fixed check order, one atomic replace per write) and safe (`0010`'s validate-before-write). Documented here rather than via a code change.
- **Ch. 6, "MUST make the exit-code contract stable and documented"**: `main()`'s own docstring previously said only "0 if no violations, 1 if violations found," understating everything else that already set `exit_code = 1` (unparseable files, unknown `--select`/`--ignore` names, no checks enabled) and omitting that `argparse` itself calls `sys.exit(2)` directly for a malformed CLI argument (bypassing `main()`'s own return entirely) — a third, undocumented value. Fixed by rewriting the docstring to state the full contract, and added `test_main_malformed_cli_argument_exits_via_argparse` to lock down the `argparse`-owned exit code `2` explicitly rather than leaving it as incidental, untested behavior a caller might not expect.

## Consequences

- `CheckOrchestrator` gains a second public attribute, `rule_failures: list[tuple[str, str]]`, populated whenever a check's `check()` or `fix()` raises unexpectedly, alongside `unprocessable_files`/`cache`/`checks`/`fix_mode`. Existing callers that only read `all_violations`/`unprocessable_files` are unaffected.
- `main()`'s exit code is now `1` (previously `0`) when a check raises during analysis or fixing and nothing else in the run would otherwise have reported it — a behavior change for that specific edge case, same category as `0011`'s equivalent change for unparseable files.
- A violation whose fix raised something other than `FixValidationError` is now reported `[FIX ERRORED]` instead of the misleading ordinary `[FIXABLE]`/"Run with --fix" pairing; `Violation.fix_data` can now carry a `fix_errored` key alongside the existing `fixed`/`fix_rejected` ones.
- A file where any check crashed is no longer cached, so a transient failure no longer freezes a false "clean" result into every subsequent cache-hit run.
