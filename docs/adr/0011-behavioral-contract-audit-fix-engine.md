# Behavioral contract audit: fix-engine write & validation safety (ch. 1, 3, 4, 20)

`docs/behavioral_contract.md` chapters **1** (Correctness and Safety of Auto-Fixes), **3** (Source File Integrity), **4** (Parsing and Invalid Python), and **20** (Parsing, AST, CST, and Source Mapping) were checked line-by-line against the fix-application and file-write path: `atomic_write_text()`/`FixValidationError` in `_base.py`, `CheckOrchestrator._apply_fixes()`/`_check_file()` in `ast_checks/__init__.py`, and every check's own `fix()` (`forbid_vars.py`, `excessive_blank_lines.py`, `misplaced_comment.py`, `redundant_assignment/autofix.py`, `validate_function_name/autofix.py`; `redundant_super_init.py` has no autofix at all, so chapters 1/20's write-path items don't apply to it).

Most items were already satisfied, several by prior ADRs (`0004` encoding/newline fidelity, `0006` typed fix data, `0010` fix validation before write) and an existing test suite that already covers BOM handling, PEP 263 encoding round-trips, CRLF preservation on untouched lines, stale-position recomputation across checks in the same run, per-violation fix rejection, and cache-corruption/missing-cache-dir handling. Idempotence (ch. 1: "MUST make each fix idempotent whenever reasonably possible") was spot-checked by running `--fix` twice over every `tests/fixtures/**/bad/*.py` fixture across all 6 checks: all 33 converged after one pass, none changed on a second pass.

## Gaps found and fixed

- **`misplaced_comment.fix()` and `redundant_assignment/autofix.apply_fixes()` let `atomic_write_text()`'s `OSError` propagate uncaught**, unlike `forbid_vars.fix()`, `excessive_blank_lines.fix()`, and `validate_function_name/autofix.apply_fix()`, which all catch `OSError` and return `False`. Calling either check's `fix()` directly (as unit tests already do, bypassing `CheckOrchestrator`) with e.g. a missing parent directory raised instead of returning `False` as the `ASTCheck.fix()` protocol documents ("Returns: True if fixes were successfully applied, False otherwise"). `CheckOrchestrator._apply_fixes`'s outer `except Exception` already masked this when running through the full pipeline, which is why it went unnoticed. Fixed by wrapping both write calls in `try/except OSError`, matching the other three checks. Regression tests: `test_fix_write_failure_returns_false` in `tests/test_misplaced_comment.py` and `tests/redundant_assignment/test_autofix.py`.
- **A file `_check_file()` couldn't read or parse vanished from the run with no trace and exit code 0** — indistinguishable from a clean file with zero violations (ch. 4: "MUST NOT silently skip a file because parsing failed", "MUST provide a clear indication when a file could not be analyzed"). `_check_file()` already logged the specific cause and returned `None` safely (no crash, other files unaffected), but nothing carried that outcome past `process_files()`'s `dict[str, list[Violation]]` return, which has no way to represent "not processed" distinctly from "processed, clean." Fixed by adding `CheckOrchestrator.unprocessable_files: list[str]`, populated whenever `_check_file()` returns `None` and reset at the start of every `process_files()` call, and having `main()` print one line per file and set `exit_code = 1` for each. Regression tests: `test_process_files_records_unprocessable_file`, `test_process_files_resets_unprocessable_files_between_calls`, `test_main_unparseable_file_returns_one` in `tests/test_orchestrator.py`.

Both gaps were about a check's own failure handling and the orchestrator's outer reporting, not about parsing invalid syntax more aggressively — no new defensive parsing was added, consistent with this project's `check-ast`/ruff precondition (see below).

## Items judged not applicable, with rationale

- **Ch. 3, concurrent external modification** ("MUST detect or safely handle concurrent source modifications where necessary", "MUST NOT silently overwrite a newer version of a file created by another process"): a fix reads a file, computes a full new content string, and writes it back; there is no compare-and-swap against the file's state at write time. Deletion between discovery and processing _is_ handled and tested (`_read_source` failure -> `None` -> now surfaced via `unprocessable_files`; `test_apply_fixes_marks_nothing_fixed`'s `file-disappears-*` cases) — only the narrower "modified, not deleted, by another process mid-run" race is unguarded. This repo has no parallel workers and no daemon; the only realistic actor is a human editing the same file in another program while `--fix` runs locally, which is the same limitation essentially every "linter with `--fix`" has. Not fixed — a real fix would need an mtime/hash compare-and-abort before the rename, which is disproportionate for a single-maintainer local hook.
- **Ch. 1, "MUST distinguish between a diagnostic that was fixed and a diagnostic that disappeared as a side effect of another fix"**: if check A's fix happens to make check B's already-collected violation vanish (rare — no two shipped checks are known to interact this way in practice), `CheckOrchestrator._apply_fixes` recomputes B's violations fresh immediately before B's own fix() call and finds nothing, so the stale violation is dropped from the report rather than mislabeled `[FIXED]`. Nothing false is ever reported, but there's also no explicit "disappeared as a side effect" marker distinguishing it from "was never a violation." Not fixed — would require snapshotting every check's original violations against a final full-file recheck, disproportionate for a project with 6 fixed, first-party checks run in a fixed order.
- **Ch. 4, syntax recovery/reporting for invalid Python generally**: this project's own precondition (AGENTS.md) is that every file it processes already passed `check-ast`/ruff in the same pre-commit run, so `ast_checks` itself never needs to parse invalid Python as a normal case. The existing catch-log-and-skip behavior in `_check_file()` (now also surfaced via `unprocessable_files`, see above) is deliberately the extent of this project's handling — it's defense-in-depth against that precondition being violated unexpectedly (e.g. the hook invoked directly, outside `.pre-commit-config.yaml`'s ordering), not a recovery path to be made more sophisticated. No new defensive parsing, syntax-error recovery, or partial-AST handling was added.
- **Ch. 20, "MUST handle tabs correctly when calculating displayed columns"**: `Violation.col` and every internal `col_offset` are used only to locate text for editing (via `byte_col_to_char_col()`'s UTF-8-byte-to-character conversion); `main()`'s reporting loop never prints a column number to the user (only `file:line: CODE: message`). There is no "displayed column" concept in this tool for tab expansion to get wrong.

## Consequences

- `CheckOrchestrator` gains a new public attribute, `unprocessable_files`, alongside the existing `cache`/`checks`/`fix_mode`. Callers that only care about violations (every existing call site) are unaffected — the dict return type and its contents are unchanged.
- `main()`'s exit code is now `1` (previously `0`) when any input file couldn't be read or parsed, even if every other file is clean. This is a behavior change for that one edge case; anything scripting around this hook that relied on the previous silent-success behavior for unparseable files was relying on undocumented, unintended behavior.
- `misplaced_comment.fix()` and `redundant_assignment/autofix.apply_fixes()` now match `forbid_vars`/`excessive_blank_lines`/`validate_function_name`'s existing `OSError`-catching convention; no protocol change.
