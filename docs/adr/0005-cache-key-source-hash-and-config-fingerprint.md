# Replace the hand-bumped CACHE_VERSION with a whole-tree source hash plus a per-check config fingerprint

`CacheManager.CACHE_VERSION` (`_cache.py:53`) is a single string a developer must remember to bump whenever any check's detection/fix logic changes in a way that could make a previously-cached result stale. Forgetting has already caused a real bug, fixed in `0e3efba` ("rewrite stale version tag when invalidating a cache blob") — that fix repaired the symptom (a stale version tag not being rewritten), not the underlying fragility of relying on a human to remember to bump a global string at all.

While designing the replacement, a second, related gap surfaced: `CheckOrchestrator._generate_cache_key()` (`ast_checks/__init__.py:185-192`) builds its cache key from the sorted `check_id` list only — it never accounts for a check's runtime configuration. `ForbidVarsCheck(forbidden_names={"custom"})` and `ForbidVarsCheck()` both produce the identical key `"forbid-vars"`, so changing `--forbid-vars-names` or `[tool.forbid-vars.autofix]` between runs on an unchanged file serves the previous run's violations under the old configuration, silently.

## Considered Options

- **Per-check `behavior_version` class attribute, folded into the cache key**: rejected as the primary mechanism — it still relies on a developer remembering to bump something, just distributed across more places instead of one. It also can't, by itself, catch a change to _shared_ code a check depends on (`_base.py`, `_scope.py`, `_cache.py`, `_prefilter.py`) unless every check's version is bumped whenever any shared module changes, which is the same discipline problem in a different shape — this is exactly the mechanism that would have missed the `0e3efba` bug, since that bug was a change to `_scope.py`, not to either affected check's own file.
- **Keep the global `CACHE_VERSION`, add a safeguard that flags missed bumps**: rejected — detecting "did behavior change without a bump" requires comparing current source against some prior fingerprint, which means computing a content hash anyway. This option is the same mechanism as the next one, with an extra manual annotation step layered on top for no added benefit.
- **Hash the whole `src/pre_commit_hooks/` source tree (excluding `tests/`) as one input to the cache key, plus a fingerprint of each enabled check's constructor arguments**: adopted. A single content hash over the whole package can't miss a shared-dependency change, since any file that could affect any check's behavior — including `_cache.py` itself, which subsumes the old "cache format changed" concern `CACHE_VERSION` also had to cover — is inside the hashed tree. The per-check config fingerprint closes the separate config-not-in-cache-key gap found above. Hashing the tree with SHA-1 (the same primitive `CacheManager.compute_file_hash` already uses) measured about 0.2ms on this machine for the current tree (19 files, 5,762 lines, 213 KB) — negligible next to the per-invocation Python interpreter startup cost `docs/adding-a-check.md` already identifies as the dominant cost at this project's size.

## Consequences

- `CACHE_VERSION` as a hand-maintained string is removed entirely; the cache key becomes: sorted `check_id` list + a fingerprint of each enabled check's constructor arguments + a SHA-1 hash of the concatenated source of every `.py` file under `src/pre_commit_hooks/`.
- The tree hash is recomputed fresh on every process invocation rather than cached to disk itself — the simplest correct option; revisit only if `scripts/benchmark.py` shows it matters.
- A change to any file in `src/pre_commit_hooks/` — including a check's own file, a shared module, or `_cache.py`'s own serialization logic — invalidates every cached result for every check, not just the ones a finer-grained scheme would isolate. Traded deliberately for eliminating the class of bug that already occurred once.
- Tracked as `docs/work-items/07-structural-cache-invalidation.md` and `docs/work-items/09-cache-key-missing-check-config.md`.
